========================================================================
STEP 1 - FIX RUNTIME READINESS FIRST
========================================================================

Objective
Make the local environment installable and testable from scratch with no
missing imports, no ambiguous commands, and no broken verification path.

Files To Review
- requirements.txt
- README.md
- docs/guides/TESTING.md
- scripts/dev_workflow.py
- start_server.py
- start_server_v3.py

Commands To Run
1. Create or refresh the local virtual environment:
   python3 -m venv venv

2. Activate it:
   source venv/bin/activate

3. Upgrade pip:
   python -m pip install --upgrade pip

4. Install project dependencies:
   python -m pip install -r requirements.txt

5. Install the Playwright browser used by UI acceptance:
   python -m playwright install chromium

6. Verify the imports that previously failed:
   python -c "import numpy, scipy, flask, requests, playwright; print('core imports ok')"

7. Verify the optional ML-related dependencies that topic/extraction code may use:
   python -c "import sklearn; print('sklearn ok')"
   python -c "import sentence_transformers; print('sentence_transformers ok')"

8. Run the main test suites:
   python test_debate_system.py
   python test_fact_check_skill.py

9. Run the combined repo check helper:
   python scripts/dev_workflow.py check

10. Start the preferred local server:
   python start_server.py --host 127.0.0.1 --port 5000

11. In a second terminal, run manual smoke tests:
   source venv/bin/activate
   python test_manual.py server-check --base-url http://127.0.0.1:5000
   python test_manual.py scenario-ai --base-url http://127.0.0.1:5000

12. Run UI acceptance on a clean temporary server:
   source venv/bin/activate
   python scripts/dev_workflow.py acceptance

What To Fix If Something Breaks
- If pip install fails:
  Confirm you are inside the project venv and that all packages in
  requirements.txt are still installable on your current Python version.
- If a ModuleNotFoundError appears:
  Add or correct the missing package in requirements.txt, then update README.md
  and docs/guides/TESTING.md so the setup path matches reality.
- If Playwright fails:
  Re-run "python -m playwright install chromium" and verify the browser binary
  is available.
- If a server start command fails:
  Confirm you are using v3, because start_server.py is a wrapper around
  start_server_v3.py.
- If acceptance fails:
  Read the generated report in artifacts/acceptance/ and inspect the matching
  screenshot(s) before changing code.

Definition Of Done
- No ModuleNotFoundError for numpy or other required packages.
- requirements.txt is sufficient for a fresh install.
- test_debate_system.py passes.
- test_fact_check_skill.py passes.
- scripts/dev_workflow.py check passes.
- scripts/dev_workflow.py acceptance passes.
- README.md and docs/guides/TESTING.md match the actual working commands.

Deliverables
- A clean, repeatable local setup flow.
- Updated docs if any commands or dependencies changed.
- Fresh acceptance artifacts in artifacts/acceptance/.

========================================================================
STEP 2 - MAKE THE ADMIN PAGE REAL
========================================================================

Objective
Turn the admin surface from a mostly presentational UI into a real moderation
configuration workflow backed by persistent data and API endpoints.

Current Problem
frontend/admin.html still contains placeholder behavior:
- saveDraft() only shows an alert
- applyTemplate() only shows confirm/alert
- loadTemplatePreview() only logs to the console
- version history and moderation outcome tables are currently hard-coded

Primary Files To Update
- frontend/admin.html
- frontend/static/js/common.js
- backend/app_v3.py
- backend/debate_engine_v2.py
- backend/database_v3.py
- backend/database.py
- backend/modulation.py

Implementation Plan
1. Define the moderation template data model.
   The stored payload should include:
   - base template id
   - version
   - status (draft or active)
   - topic requirements
   - toxicity settings
   - PII settings
   - spam/rate limit settings
   - prompt injection settings
   - created_at
   - updated_at
   - optional author/admin id

2. Add database persistence.
   Create one or more tables for:
   - saved drafts
   - versioned applied templates
   - active template pointer

3. Add backend endpoints in backend/app_v3.py.
   Suggested endpoints:
   - GET /api/admin/moderation-template/current
   - GET /api/admin/moderation-template/history
   - POST /api/admin/moderation-template/draft
   - POST /api/admin/moderation-template/apply

4. Add the corresponding database methods.
   The API layer should not directly manage raw SQL for everything.
   Put read/write helpers in the database layer.

5. Make the debate engine consume the active template.
   When a template is applied, the next snapshot and future post moderation
   should use that saved active template rather than a hard-coded default only.

6. Replace hard-coded admin numbers with live data.
   The following should come from real backend responses:
   - current template name/version
   - version history
   - allowed/blocked counts
   - block reasons histogram

7. Replace placeholder JS behavior in frontend/admin.html.
   saveDraft() should call the draft endpoint.
   applyTemplate() should call the apply endpoint.
   loadTemplatePreview() should load real template state into the form and
   preview panels.

8. Add loading, success, and error states.
   Do not leave admin actions silent. The user should see:
   - save in progress
   - apply in progress
   - save/apply succeeded
   - failure with actionable error text

9. Decide the permissions model.
   At minimum, decide whether:
   - any authenticated user can use admin
   - only a configured admin user can use admin
   - admin is temporarily open in local dev but restricted later
   Encode that decision in the API instead of relying on the page label.

Testing Checklist
- Open admin.html and load the current template from the backend.
- Change one field and save a draft.
- Reload the page and confirm the draft persisted.
- Apply the template and confirm version history updates.
- Generate a new snapshot from new_debate.html and confirm the new template
  name/version appears on the snapshot page.
- Confirm admin history and moderation outcome sections no longer show only
  hard-coded example rows.

Definition Of Done
- No alert()/confirm()-only admin actions remain.
- Admin form state persists through the backend.
- Applied template versions are stored and listed in history.
- Snapshot pages reflect the actually active moderation template.
- Admin page is backed by live data, not static demo content.

Deliverables
- Real admin CRUD flow for moderation templates.
- Persisted version history.
- Live moderation statistics in the admin surface.
