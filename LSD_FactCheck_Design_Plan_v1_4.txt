1. EXECUTIVE SUMMARY
============================================================

This document specifies an LLM-based empirical fact-checking subsystem designed to fulfill the LSD v1.2 requirements (§13). The system does not attempt to be a "single truth oracle" but rather a structured, auditable pipeline that classifies empirical premises into:

- SUPPORTED (p = 1.0)
- REFUTED (p = 0.0)
- INSUFFICIENT (p = 0.5)

with explicit evidence tiers, citations, operationalization records, and insufficiency reasons.

CRITICAL DESIGN DECISIONS:
- The system operates on immutable snapshots. "Live" connectors are queried at snapshot creation time, and their outputs are frozen into the snapshot. This preserves auditability (§4) and artifact replay (§4 replay manifest).
- The synthesis engine produces authoritative outputs via hard-coded rules and structured proof traces. LLM-generated prose is available only as a non-authoritative display summary.
- Artifact replay (reconstruction from frozen records) is the authoritative audit mode. Computational reruns are diagnostic only.
- All scored premises preserve provenance span citations (§7) from upstream AU extraction through to final dossier outputs.
- Insufficiency sensitivity and decisive premise impact are computed as first-class outputs to feed §17 "what would change this" and §13.1 sensitivity analysis.

============================================================
2. DESIGN PRINCIPLES & CONSTRAINTS
============================================================

2.1 Derived from LSD v1.2
- Deterministic selection (§11): The *selection* of which premises to check is deterministic. The *checking process* must be auditable. The published snapshot must be deterministically replayable from frozen artifacts.
- Evidence tiers (§13): Must publish Tier 1/2/3 counts per topic-side.
- Insufficiency handling (§13.1): Must emit INSUFFICIENT with p=0.5 when evidence is truly unavailable or undecidable; must publish sensitivity of INSUFFICIENT assumptions on overall scores.
- Snapshot immutability (§4): All fact-check outputs are frozen with tamper-evident hashes.
- Perfect checker contract (§13): If sufficient evidence exists, classify SUPPORTED/REFUTED with p ∈ {1,0}. If evidence is unavailable, p=0.5.
- Traceability (§7): All scored premises MUST cite provenance spans.

2.2 Core Architectural Principles
- Separation of concerns: truth status (p) vs. confidence vs. evidence quality
- Wikidata as canonicalization, not truth
- Evidence policy governs what sources are acceptable for what claim types
- All connectors return EvidenceItems; tiers are assigned per-item, not per-connector
- Compound claims must be decomposed before routing, preserving logical structure as a ClaimExpression tree
- Evidence verdicts must be scoped: date, geography, measurement definition, source basis, and rounding tolerance
- Human review is a first-class diagnostic output for ambiguity, policy gaps, contradictory high-tier evidence, and high-impact insufficiency
- Authoritative outputs are structured and rule-generated; LLM prose is display-only
- Provenance spans flow through every stage from intake to dossier output

2.3 Replay Semantics
- Artifact replay: Reconstruct the published snapshot from frozen stored inputs, evidence records, audit records, and output hashes. This mode MUST be deterministic and is the basis for all public audit claims.
- Computational rerun: Re-run connectors and LLM steps from the same replay manifest. This mode is diagnostic only and MAY differ because external APIs, live sources, and hosted model behavior can change unless all connector responses and model outputs are pinned.
- Public audit claims should be based on artifact replay. Computational reruns are used for drift detection, regression testing, and connector health checks.

4.1 CLAIM DECOMPOSER + CLAIM TYPE CLASSIFIER
------------------------------------------------------------

PURPOSE:
Transform natural language empirical premises into atomic, checkable subclaims, classify their epistemic type, and preserve the original logical structure as a ClaimExpression tree with provenance spans.

INPUT:
- Canonical empirical premise text (from §8 AU extraction)
- Provenance spans from upstream (span_id, post_id, offsets, span_text)
- Topic context
- Frame context (for disambiguation)

OUTPUT:
List of AtomicSubclaim objects:
  {
    subclaim_id: UUID,
    text: string,
    claim_type: ClaimType,
    secondary_claim_types: list[ClaimType],
    claim_expression: ClaimExpression,
    operationalization_hint: string,
    verdict_scope_hint: {
      temporal_scope: string | null,
      geographic_scope: string | null,
      population_scope: string | null,
      measurement_definition: string | null,
      source_basis: string | null,
      rounding_tolerance: string | null
    },
    parent_premise_id: string,
    provenance_spans: list[{span_id, post_id, offsets, span_text}],
    decomposition_rationale: string
  }

CLAIM EXPRESSION SCHEMA:
  {
    node_type: enum {ATOMIC, AND, OR, NOT, IF_THEN, COMPARISON, QUANTIFIER},
    children: list[ClaimExpression],
    subclaim_id: string | null,        // required when node_type == ATOMIC
    operator: string | null,            // e.g., ">", "<", "more_than", "equal_to" for COMPARISON
    quantifier: string | null,          // e.g., "MOST", "AT_LEAST_N", "EXISTS", "MAJORITY" for QUANTIFIER
    quantifier_parameter: string | number | null,  // e.g., N for AT_LEAST_N
    comparison_target: string | null     // for COMPARISON nodes
  }

EVALUATION RULES FOR ClaimExpression:
- ATOMIC: leaf node referencing a subclaim_id. Evaluated by the synthesis engine using that subclaim's result.
- AND: All children must evaluate to SUPPORTED for the node to be SUPPORTED. Any REFUTED child makes the node REFUTED. Any INSUFFICIENT child (with no REFUTED) makes the node INSUFFICIENT.
- OR: Any SUPPORTED child makes the node SUPPORTED. All children REFUTED makes the node REFUTED. Any INSUFFICIENT child (with no SUPPORTED) makes the node INSUFFICIENT.
- NOT: Inverts the child's status. SUPPORTED → REFUTED, REFUTED → SUPPORTED, INSUFFICIENT → INSUFFICIENT.
- IF_THEN: Evaluates antecedent (children[0]) first. If antecedent is REFUTED → node is INSUFFICIENT (see §4.6 conditional rule). If antecedent is INSUFFICIENT → node is INSUFFICIENT. If antecedent is SUPPORTED, evaluate consequent (children[1]).
- COMPARISON: Evaluates whether a quantitative relationship holds between two compared subclaims or values. Requires both operands to be SUPPORTED or REFUTED with comparable verdict_scope.
- QUANTIFIER: Evaluates whether a quantified claim holds over a set. Requires the quantified set to be resolved and the predicate to be checkable.

Max recursion depth: 3. Expressions deeper than 3 are rejected and flagged as POLICY_GAP.

PROVENANCE SPAN RULE:
All provenance spans from the parent premise MUST be preserved in every AtomicSubclaim output. The fact-check result will carry these spans forward so that §7 traceability and §17 decisive premises/arguments list with spans are satisfied. If a decomposition creates subclaims that only use a subset of spans, the full span set is retained at the premise level; subclaims may reference the specific span(s) they derive from.

CLAIM TYPE ENUM (with priority order for multi-type claims):
  LEGAL_REGULATORY      → highest burden
  SCIENTIFIC            → requires peer review or institutional backing
  CAUSAL                → requires mechanism or strong association
  CURRENT_EVENT         → requires recent, verified reporting
  NUMERIC_STATISTICAL   → requires official statistics
  TEMPORAL              → date/event verification
  GEOGRAPHIC_DEMOGRAPHIC → structured KB or census
  EMPIRICAL_ATOMIC      → directly checkable
  EMPIRICAL_COMPOUND    → requires decomposition
  PREDICTIVE            → future-oriented; NOT_FACT_CHECKABLE for p
  NORMATIVE             → value judgment; route to §14
  AMBIGUOUS             → cannot operationalize
  NOT_FACT_CHECKABLE    → not empirical

CLAIM TYPE PRIORITY RULE:
If a subclaim matches multiple types, the decomposer selects the type with the highest evidentiary burden (top of the priority list above). The full list of applicable types is stored in secondary_claim_types. The Evidence Policy module uses the primary claim_type for source selection.

VALIDATION LAYER (rule-based guardrails):
- Reject decompositions that introduce new claims not in premise
- Reject decompositions that change logical structure (ClaimExpression must be semantically equivalent to the original premise)
- Reject operationalization_hints that conflict with Evidence Policy verdict_scope_requirements (flag POLICY_GAP)
- Cross-check verdict_scope_hint against EvidencePolicy.verdict_scope_requirements; reject mismatches with flag POLICY_GAP
- Max recursion depth: 3

AUDIT OUTPUT:
- Full prompt template version
- LLM model version, temperature, seed
- Raw LLM output before validation
- Validation pass/fail log

4.2 ENTITY / CONCEPT LINKING LAYER
------------------------------------------------------------

PURPOSE:
Canonicalize entities and concepts to stable identifiers for cross-snapshot consistency and cross-language support.

INPUT:
- Atomic subclaim text
- Topic context

OUTPUT:
EntityLink objects:
  {
    entity_id: UUID,
    mention_span: {start, end, text},
    canonical_id: string,
    canonical_type: enum {WIKIDATA, DOI, PUBMED, LEGAL_CITATION, DATASET_ID, UNRESOLVED},
    canonical_label: string,
    linking_confidence: float [0,1],
    ambiguity_flag: boolean,
    ambiguity_candidates: list
  }

WIKIDATA INTEGRATION:
- Use Wikidata API (wbsearchentities) for initial candidate generation
- Use LLM disambiguation with context window
- Store QIDs for: persons, organizations, locations, events, concepts
- Store PIDs for: relations/properties when relevant
- Wikidata is NOT used for truth verification at this stage

OTHER IDENTIFIERS:
- DOI / PubMed ID: Extracted via regex + Crossref/PubMed API
- Legal citations: Extracted via citation parser + court API lookup
- Government dataset IDs: Matched against known dataset registries
- Custom dataset: Maintain mapping table of common debate entities

UNRESOLVED ENTITIES:
- If linking confidence < 0.7 or ambiguity_candidates > 1
- Flag as AMBIGUOUS in subclaim
- Route to INSUFFICIENT with reason "entity_resolution_failure"

AUDIT OUTPUT:
- Linking query parameters
- API responses (truncated if large)
- Disambiguation rationale
- Confidence scores

4.3 EVIDENCE POLICY MODULE
------------------------------------------------------------

PURPOSE:
Governs what source types are acceptable for what claim types, preventing arbitrary synthesis and ensuring consistent standards.

INPUT:
- Atomic subclaim
- Claim type (primary, from priority rule)
- Entity links
- Available connector inventory

OUTPUT:
EvidencePolicy object:
  {
    policy_id: string,
    claim_type: ClaimType,
    required_source_types: list[SourceType],
    preferred_source_types: list[SourceType],
    minimum_acceptable_tier: int,      // Tier N or stronger (lower number = stronger). Tier 1 and Tier 2 both satisfy minimum_acceptable_tier: 2.
    cross_verification_required: boolean,
    cross_verification_minimum_sources: int,
    temporal_constraint: {max_age_days, require_contemporary} | null,
    verdict_scope_requirements: list[string],
    frame_dependent: boolean,          // true only if policy scope depends on Frame
    special_rules: list[string]
  }

DEFAULT POLICY RULES:

For NUMERIC_STATISTICAL claims:
  - Required: official statistical agency or authoritative dataset API
  - Preferred: BLS, Census Bureau, Eurostat, UN databases, World Bank
  - Minimum acceptable tier: 2 (Tier 1 or Tier 2 satisfy)
  - Cross-verification: false

For LEGAL_REGULATORY claims:
  - Required: official legal/government source
  - Preferred: Government gazette, official legislative database, court records
  - Minimum acceptable tier: 1 (only Tier 1 satisfies)
  - Cross-verification: false

For SCIENTIFIC claims:
  - Required: peer-reviewed source OR major scientific institution statement
  - Preferred: Systematic review, meta-analysis, primary paper
  - Minimum acceptable tier: 2 (Tier 1 or Tier 2 satisfy)
  - Cross-verification: true for causal, consensus, safety, efficacy, or generalized claims
  - Scope rule:
    * Study-specific claim: a primary peer-reviewed paper may be Tier 1 evidence.
    * Consensus/generalized claim: prefer systematic reviews, meta-analyses, major scientific bodies, or multiple independent studies.
    * A single study cannot alone establish broad consensus unless the claim is explicitly scoped to that study.

For GEOGRAPHIC_DEMOGRAPHIC claims:
  - Required: structured KB or official statistical source
  - Preferred: Wikidata (if referenced and undisputed) + census data
  - Minimum acceptable tier: 2 (Tier 1 or Tier 2 satisfy)
  - Cross-verification: false

For CURRENT_EVENT claims:
  - Required: official confirmation OR at least 2 independent reputable Tier 2 sources
  - Preferred: Major news agencies, official government statements
  - Minimum acceptable tier for SUPPORTED/REFUTED verdict: 2
  - Tier 3 live search may discover candidate sources but cannot alone satisfy verdict evidence
  - Cross-verification: true (minimum 2 independent sources unless official confirmation exists)
  - Temporal constraint: configurable per snapshot (default max_age_days = 7, require_contemporary = true)

For CAUSAL claims:
  - Required: evidence of mechanism OR strong statistical association
  - Preferred: Peer-reviewed studies, official evaluations
  - Minimum acceptable tier: 2 (Tier 1 or Tier 2 satisfy)
  - Cross-verification: true (minimum 2 sources)

POLICY VERSIONING:
- Policies are versioned and published per snapshot (§4)
- Changelog tracked (§20)
- Emergency policy changes follow §20.2 incident protocol
- Temporal constraints (e.g., max_age_days for CURRENT_EVENT) are configurable per snapshot, not hard-coded

4.4 EVIDENCE ROUTER
------------------------------------------------------------

PURPOSE:
Retrieve evidence from appropriate backends based on evidence policy, claim type, and entity links.

CONNECTORS:

A) TIER 1 OFFICIAL / PRIMARY CONNECTORS
   - Government statistics APIs (BLS, Census, Eurostat, etc.)
   - Scientific databases (PubMed, arXiv, DOI resolvers)
   - Legal databases (court APIs, legislation APIs)
   - Official organization APIs (UN, WHO, World Bank)
   - Custom dataset connectors for known debate domains

B) STRUCTURED KB CONNECTOR (WIKIDATA)
   - SPARQL endpoint queries using linked QIDs/PIDs
   - Used for: static factual anchors (dates, geography, demographics, org relationships)
   - NOT used for: causal claims, current events, policy evaluations, predictive claims
   - EvidenceItem tier assignment depends on reference quality:
     * Wikidata statement with no reference → Tier 3
     * Wikidata statement with secondary/reputable reference → Tier 2
     * Wikidata statement with primary/official reference → treat Wikidata as the retrieval path to the primary source, not as the primary source itself. The resulting EvidenceItem should use the underlying source_type when retrievable, with retrieval_path = WIKIDATA_REFERENCE.
   - Absence of a Wikidata statement normally means INSUFFICIENT, not REFUTED, unless an evidence policy explicitly defines a complete closed-world source for the claim.

C) TIER 2 CURATED RAG
   - Vector database of: Wikipedia, curated news archives, academic review papers, reputable secondary syntheses
   - Retrieval: Dense passage retrieval (DPR) or similar
   - Re-ranking: Cross-encoder for relevance
   - Max retrieved documents: 10 per subclaim
   - Date filter: Exclude sources older than policy temporal constraint

D) TIER 3 LIVE SEARCH
   - Web search APIs (Brave, Bing, Google Custom Search)
   - News APIs (NewsAPI, GDELT)
   - Used for: source discovery, freshness checks, unresolved claims where Tier 1/2 insufficient
   - Evidence from Tier 3 CANNOT alone support SUPPORTED/REFUTED
   - Tier 3 evidence can:
     * Support INSUFFICIENT verdict
     * Be promoted to Tier 2 via Source Reputation Registry (see below)
     * Provide context for operationalization

SOURCE REPUTATION REGISTRY (Tier 3 Promotion Mechanism):
- Maintain an allowlist of domains/publications that qualify as Tier 2
- Tier 3 discovery → check URL against registry
- If matched:
  * Attempt re-fetch via direct API if available
  * If no API, store page content hash + archive.org permalink as EvidenceItem
  * Assign Tier 2 only if independence checks pass (distinct from other sources)
- Promotion requires: (a) registry match AND (b) independence verification AND (c) policy approval
- Registry is versioned and published per snapshot

SOURCE INDEPENDENCE DEFINITION (for Tier 3 promotion and cross-verification):
Independent sources must not be:
- Syndicated copies of the same wire article
- Articles citing only the same original report
- Same publisher group without editorial independence
- Rewritten versions of the same press release
- Sources whose only basis is each other
- Bot-generated or content-farm variants of the same underlying text

ROUTING LOGIC:
1. Query all applicable connectors in parallel
2. Apply policy filters (source type, temporal constraints)
3. Collect EvidenceItems
4. If minimum acceptable tier not met by any item → early INSUFFICIENT flag
5. If cross-verification required but insufficient independent sources → INSUFFICIENT

CONNECTOR VERSIONING:
- Each connector has version string
- API endpoint URLs recorded
- Query parameters hashed
- Response metadata stored (timestamp, status code, latency)

CONNECTOR FALLBACK CHAIN:
- Configurable per connector type in a published fallback table
- Example: BLS API → Census API → Tier 2 RAG (for statistical claims)
- Example: CourtListener → PACER → Tier 2 legal RAG (for legal claims)
- Fallback triggers: HTTP error, timeout, schema mismatch, empty result set

4.5 EVIDENCE NORMALIZER
------------------------------------------------------------

PURPOSE:
Transform heterogeneous connector outputs into standardized EvidenceItem objects for uniform processing.

EVIDENCE ITEM SCHEMA:
  {
    evidence_id: UUID,
    subclaim_id: string,
    source_type: enum {OFFICIAL_STAT, GOV_DB, SCIENTIFIC_DB, LEGAL_DB, WIKIDATA, WIKIPEDIA, NEWS, WEB, OTHER},
    source_tier: int,
    retrieval_path: enum {DIRECT_CONNECTOR, WIKIDATA_REFERENCE, RAG_RETRIEVAL, LIVE_SEARCH_DISCOVERY, MANUAL_UPLOAD},
    source_url: string,
    source_title: string,
    source_date: ISO8601,
    source_authority: string,
    quote_or_span: string,
    quote_context: string,
    verdict_scope: {
      temporal_scope: string | null,
      geographic_scope: string | null,
      population_scope: string | null,
      measurement_definition: string | null,
      source_basis: string | null,
      rounding_tolerance: string | null
    },
    relevance_score: float [0,1],
    direction: enum {SUPPORTS, REFUTES, UNCLEAR, NEUTRAL},
    direction_confidence: float [0,1],
    direction_method: enum {DETERMINISTIC_STRUCTURED, LLM_CLASSIFIER, MANUAL},
    retrieval_timestamp: ISO8601,
    connector_version: string,
    connector_query_hash: string,
    source_snapshot_id: string | null,
    raw_response_hash: string
  }

NORMALIZATION RULES:
- Truncate quotes to 1000 chars max
- Standardize dates to ISO8601
- Extract authority from source metadata (publisher, organization)
- Compute relevance_score via LLM + cross-encoder ensemble
- Direction classification:
  * For structured sources (APIs, databases, official stats): use deterministic comparison logic whenever possible (claimed_value == source_value → SUPPORTS; claimed_value != source_value under closed-world policy → REFUTES; no comparable value → UNCLEAR). Set direction_method = DETERMINISTIC_STRUCTURED.
  * For text/RAG/web sources: use LLM classifier with few-shot examples. Set direction_method = LLM_CLASSIFIER.
  * If direction_confidence < 0.7 → direction = UNCLEAR
  * An UNCLEAR item cannot alone determine SUPPORTED or REFUTED for its subclaim
- Reject items with relevance_score < 0.3
- Preserve retrieval_path so primary evidence discovered through Wikidata is attributed to the primary source
- Extract verdict_scope whenever the source defines date, jurisdiction, population, metric definition, or rounding rules

SOURCE_SNAPSHOT_ID RULES (HARDENED):
- For static dumps (Wikidata, Wikipedia): use dump date / dump ID
- For live APIs: use API version + response timestamp + endpoint URL
- For web pages (Tier 3 or any URL-based source):
  * MUST store content hash (SHA-256 of normalized page text)
  * MUST attempt archive.org permalink generation
  * If archive.org fails, MUST store full page content in snapshot storage
  * The archived content or permalink is required for artifact replay verification
- For all sources: raw_response_hash is mandatory

4.6 POLICY-GATED SYNTHESIS ENGINE
------------------------------------------------------------

PURPOSE:
Synthesize normalized evidence into a final fact-check result per subclaim via hard-coded rules, then aggregate to premise level using the ClaimExpression tree.

DESIGN PRINCIPLE:
The synthesis engine is rule-driven. The LLM does not generate the authoritative verdict, rationale, or status. The LLM may be used in downstream display summary generation (§4.10), which is explicitly non-authoritative.

INPUT:
- Atomic subclaim
- EvidencePolicy
- List[EvidenceItem]
- Entity links

OUTPUT:
SubclaimResult:
  {
    subclaim_id: string,
    status: enum {SUPPORTED, REFUTED, INSUFFICIENT},
    p: float,
    confidence: float [0,1],
    best_evidence_tier: int | null,
    limiting_evidence_tier: int | null,
    decisive_evidence_tier: int | null,
    citations: list[EvidenceItem.evidence_id],
    operationalization: string,
    verdict_scope: {
      temporal_scope: string | null,
      geographic_scope: string | null,
      population_scope: string | null,
      measurement_definition: string | null,
      source_basis: string | null,
      rounding_tolerance: string | null
    },
    insufficiency_reason: string | null,
    human_review_flags: list[HumanReviewFlag],
    provenance_spans: list[{span_id, post_id, offsets, span_text}],
    synthesis_logic: SynthesisLogic,
    synthesis_rule_engine_version: string
  }

SYNTHESIS LOGIC SCHEMA (authoritative, structured):
  {
    status_rule_applied: string,
    policy_rule_id: string,
    decisive_evidence: list[string],
    contradictory_evidence: list[string],
    subclaim_results: list[{
      subclaim_id: string,
      status: enum,
      decisive_evidence: list[string]
    }],
    verdict_scope_applied: {
      temporal_scope: string | null,
      geographic_scope: string | null,
      population_scope: string | null,
      measurement_definition: string | null,
      source_basis: string | null,
      rounding_tolerance: string | null
    },
    insufficiency_trigger: string | null,
    human_review_flags: list[string],
    authority_ranking_applied: boolean,
    claim_expression_node_type: enum {ATOMIC, AND, OR, NOT, IF_THEN, COMPARISON, QUANTIFIER}
  }

SYNTHESIS RULES (HARD CODED):

IF any Tier 1 evidence directly supports/refutes:
  AND no contradictory Tier 1 evidence:
  → SUPPORTED/REFUTED, p=1.0/0.0, confidence=high

IF contradictory Tier 1 evidence exists:
  IF one source clearly outranks the other per evidence policy hierarchy
    (e.g., Supreme Court ruling > agency preliminary report;
     final official statistics > preliminary estimate;
     systematic review > single study):
    → Follow higher-authority source: SUPPORTED/REFUTED, p=1.0/0.0
    → synthesis_logic.authority_ranking_applied = true
  ELSE:
    → INSUFFICIENT, p=0.5
    → human_review_flags includes CONTRADICTORY_TIER1_EVIDENCE

IF only Tier 2 evidence:
  AND cross-verification satisfied (if required):
  → SUPPORTED/REFUTED, p=1.0/0.0, confidence=medium

IF Tier 2 evidence mixed or insufficient:
  → INSUFFICIENT, p=0.5, confidence=medium

IF only Tier 3 evidence:
  → INSUFFICIENT, p=0.5, confidence=low

IF no evidence retrieved:
  → INSUFFICIENT, p=0.5, confidence=low
  insufficiency_reason = "no_evidence_retrieved"

IF entity resolution failed:
  → INSUFFICIENT, p=0.5
  insufficiency_reason = "entity_resolution_failure"

IF claim type is PREDICTIVE:
  → INSUFFICIENT, p=0.5
  insufficiency_reason = "predictive_claim_not_checkable"

IF evidence verdict_scope is narrower than premise/subclaim verdict_scope:
  → INSUFFICIENT, p=0.5
  insufficiency_reason = "evidence_scope_narrower_than_claim"
  human_review_flags += SCOPE_MISMATCH

CONFIDENCE CALCULATION:
confidence = f(evidence_tier, evidence_count, cross_verification_met, entity_resolution_quality, direction_consistency)

Where:
- Tier 1 + consistent + resolved → confidence ≥ 0.9
- Tier 2 + consistent + resolved → confidence 0.7-0.9
- Tier 3 or mixed → confidence ≤ 0.6
- Entity ambiguity → confidence capped at 0.5

For implementation, f() is defined as a weighted formula published per snapshot:
  base = 1.0 if best_evidence_tier == 1 else 0.8 if best_evidence_tier == 2 else 0.5
  cross_penalty = 0.2 if (cross_verification_required and not cross_verification_met) else 0.0
  entity_penalty = 0.0 if entity_resolution_quality == 1.0 else 0.3
  direction_penalty = 0.0 if direction_consistency == 1.0 else 0.2
  confidence = max(0.0, base - cross_penalty - entity_penalty - direction_penalty)

AGGREGATION TO PREMISE LEVEL:
The synthesis engine evaluates the parent premise's ClaimExpression tree recursively.

ATOMIC:
- Use the subclaim's status, p, confidence, and tiers directly.

AND:
- If any child is REFUTED → node REFUTED, p=0.0
- If all children are SUPPORTED → node SUPPORTED, p=1.0
- If any child is INSUFFICIENT and none are REFUTED → node INSUFFICIENT, p=0.5

OR:
- If any child is SUPPORTED → node SUPPORTED, p=1.0
- If all children are REFUTED → node REFUTED, p=0.0
- If any child is INSUFFICIENT and none are SUPPORTED → node INSUFFICIENT, p=0.5

NOT:
- Inverts the child's status. SUPPORTED → REFUTED, REFUTED → SUPPORTED, INSUFFICIENT → INSUFFICIENT.

IF_THEN (If A then B):
- If A is REFUTED → node INSUFFICIENT, p=0.5, insufficiency_reason="antecedent_refuted_conditional_not_substantively_checkable"
- If A is INSUFFICIENT → node INSUFFICIENT, p=0.5
- If A is SUPPORTED and B is SUPPORTED → node SUPPORTED, p=1.0
- If A is SUPPORTED and B is REFUTED → node REFUTED, p=0.0
- If A is SUPPORTED and B is INSUFFICIENT → node INSUFFICIENT, p=0.5

COMPARISON:
- Both operands must be resolved (SUPPORTED or REFUTED) with comparable verdict_scope.
- If either operand is INSUFFICIENT → node INSUFFICIENT, p=0.5
- If comparison holds per deterministic evaluation → node SUPPORTED, p=1.0
- If comparison fails → node REFUTED, p=0.0

QUANTIFIER:
- If the quantified set or predicate is INSUFFICIENT → node INSUFFICIENT, p=0.5
- Evaluate against the threshold/rule defined in the quantifier node.

For all structures:
- Confidence = mean(child confidences) for AND/OR; child confidence for NOT/IF_THEN/COMPARISON/QUANTIFIER
- best_evidence_tier = min(child tiers) for AND/OR; child tier for NOT/IF_THEN/COMPARISON/QUANTIFIER
- limiting_evidence_tier = max(child tiers)
- decisive_evidence_tier = tier of the child or evidence path that determined the node status
- Citations = union of all child citations
- provenance_spans = union of all child provenance spans (preserved from upstream §7)

TIER REPORTING RULE (COMPOUND PREMISES):
When a compound premise evaluates to INSUFFICIENT, best_evidence_tier MUST be set to null OR equal to limiting_evidence_tier. Do not report min(child tiers) for INSUFFICIENT compounds, as this would misleadingly suggest strong evidence exists while hiding the limiting subclaim. decisive_evidence_tier and limiting_evidence_tier are always reported.

4.8 CLAIM IDENTITY, CACHE, AND IMMUTABILITY RULES
------------------------------------------------------------

PURPOSE:
Ensure stable claim identity, cache reuse, and immutability across snapshots and connector/policy changes.

NORMALIZATION:
- Each atomic subclaim receives normalized_subclaim_text.
- claim_hash = SHA256(lowercase(trimmed_text_with_standardized_numbers)).
- Stopwords are NOT removed.
- Standardized numbers include consistent formatting for commas, decimals, percentages, currency, and common date formats where this does not change meaning.

CACHE KEY:
  {
    claim_hash: string,
    fact_mode: enum {OFFLINE, ONLINE_ALLOWLIST, PERFECT_CHECKER, LIVE_CONNECTORS},
    evidence_policy_version: string,
    connector_snapshot_versions: dict[connector_name -> version_or_snapshot_id],
    frame_dependency_key: null | {frame_set_version, frame_id, frame_scope_hash}
  }

FRAME DEPENDENCY RULE:
- frame_dependency_key is null unless the evidence policy explicitly declares frame_dependent = true.
- If frame_dependent = false (the default for all empirical fact checks), changing a normative Frame does not invalidate the cached empirical fact check.
- If frame_dependent = true, the key includes the Frame version and scope hash so that frame-scoped policies recompute correctly.

IMMUTABILITY RULES:
- Cached fact checks are immutable within a snapshot.
- A cached result may be reused only if the full cache key matches.
- Recompute only on explicit invalidation, evidence policy change, connector/source snapshot change, or material bug/incident correction.
- DebateFrame changes do not recompute empirical fact checks unless the evidence policy explicitly depends on Frame scope.
- All invalidations are additive: old records remain available, and the new snapshot references the replacement record.

INVALIDATION AUTHORITY:
- AUTOMATED: Connector health monitor detects API schema change or persistent failure
- GOVERNANCE: Published rule change or incident declaration (§20.2)
- APPEAL: Via §20 appeal pathway (moderation errors, provenance errors, missing key arguments)
- All invalidations are logged with timestamp, authority type, and rationale

4.9 HUMAN REVIEW QUEUE
------------------------------------------------------------

PURPOSE:
Surface high-risk or underspecified cases rather than silently forcing a brittle automated verdict.

HumanReviewFlag enum:
  NONE
  ENTITY_AMBIGUITY
  POLICY_GAP
  CONTRADICTORY_TIER1_EVIDENCE
  HIGH_IMPACT_INSUFFICIENT
  CAUSAL_COMPLEXITY
  SCIENTIFIC_SCOPE_OVERCLAIM
  LLM_VALIDATION_FAILURE
  CONNECTOR_FAILURE
  TEMPORAL_SCOPE_AMBIGUITY
  SOURCE_CONFLICT
  SCOPE_MISMATCH

QUEUE RULES:
- Any flag other than NONE adds the premise/subclaim to a review queue.
- Review queue inclusion does not change p by itself. It changes diagnostics and governance visibility.
- High-impact insufficient claims should be listed in the LSD "what would change this" output (§17).
- Public outputs report aggregate counts by flag with small-count suppression.
- Internal outputs preserve full records for governance review and incident response.

HUMAN REVIEW RECORD (additive, immutable):
When a human reviewer examines a queued item, the resolution is stored as a separate HumanReviewRecord:

  {
    review_id: UUID,
    target_audit_id: string,
    target_snapshot_id: string,
    reviewer_role: string,
    review_outcome: enum {REVIEWED_NO_CHANGE, REVIEWED_CORRECTION, REVIEWED_POLICY_GAP, REVIEWED_SOURCE_DISPUTE},
    review_note: string,
    review_timestamp: ISO8601,
    review_record_hash: string
  }

HUMAN REVIEW OUTCOMES:
1. REVIEWED_NO_CHANGE
   - Human confirms the automated result is correct.
   - Action: Publish HumanReviewRecord in audit trail. Original audit record remains frozen.
   - p remains unchanged.

2. REVIEWED_CORRECTION
   - Human identifies a bug, incident, or material error in the automated pipeline.
   - Action: Trigger §20.2 INCIDENT snapshot. Publish incident report.
   - New snapshot contains additive correction only. Old snapshot remains hash-addressable.
   - p may change in the new snapshot, but old snapshot records are never altered.

3. REVIEWED_POLICY_GAP
   - Human determines the evidence policy does not cover this claim type or source combination.
   - Action: Submit evidence policy petition for next review window (analogous to §5.3 Frame Petition Track).
   - Current snapshot p remains unchanged (typically INSUFFICIENT).
   - Policy change, if approved, applies to future snapshots only.

4. REVIEWED_SOURCE_DISPUTE
   - Human disputes the authority ranking of contradictory Tier 1 evidence.
   - Action: Flag for evidence policy hierarchy revision. Publish dispute note.
   - Current snapshot p remains unchanged.

No human review outcome permits silent retroactive modification of a published snapshot or its AuditRecord.

4.10 DISPLAY SUMMARY GENERATOR (NON-AUTHORITATIVE)
------------------------------------------------------------

PURPOSE:
Produce human-readable prose explaining the fact-check result, generated from the authoritative structured synthesis_logic. This is for public communication only and is not part of scoring, auditing, or replay.

INPUT:
- SubclaimResult.synthesis_logic
- SubclaimResult.verdict_scope
- EvidenceItem list (for citation text)

OUTPUT:
DisplaySummary:
  {
    text: string,
    generated_by: string,
    generation_timestamp: ISO8601,
    authoritative_source: "synthesis_logic",
    consistency_check: enum {PASS, FAIL, NOT_RUN},
    consistency_check_rationale: string | null
  }

GENERATION RULES:
- The LLM fills a template derived from synthesis_logic, not inventing facts.
- The summary must reference evidence by ID and include scope qualifiers (date, geography, measurement).
- The summary is explicitly labeled as non-authoritative in all publications.

CONSISTENCY CHECK:
A lightweight rule-based checker verifies the display summary does not contradict:
- status (e.g., summary says "supported" when status is REFUTED)
- p value
- decisive_evidence_tier
- insufficiency_reason

If consistency check FAILS:
- Flag LLM_VALIDATION_FAILURE
- Add to human review queue
- Discard the display summary
- Fall back to a mechanically generated template string from synthesis_logic
- Publish the fallback with label "machine-generated summary"
- The failed DisplaySummary is still stored in the audit record for completeness but is excluded from the authoritative Merkle root.

AUDIT TREATMENT:
- DisplaySummary is stored in the audit record for completeness.
- It is NOT included in the authoritative_result_hash or the snapshot Merkle root.
- Artifact replay can reconstruct the snapshot without regenerating the display summary.
- display_summary_hash is computed separately and stored for internal traceability only.

4.11 INSUFFICIENCY SENSITIVITY ENGINE
------------------------------------------------------------

PURPOSE:
Fulfill §13.1 requirement to publish sensitivity of INSUFFICIENT premises on overall scores: "If INSUFFICIENT assumed true/false, D changes by …"

INPUT:
- FactCheckResult for one premise
- All other FactCheckResults in the same (topic, side)
- Frame parameters (Rel_t weights, component combination rules from §16)
- Replicate parameters if computing per-replicate sensitivity

OUTPUT:
InsufficiencySensitivity:
  {
    premise_id: string,
    topic_id: string,
    side: enum {FOR, AGAINST},
    frame_id: string | null,
    baseline_p: float,
    baseline_F_ts: float,
    baseline_D_Frame: float | null,
    counterfactual_p_true: 1.0,
    counterfactual_p_false: 0.0,
    counterfactual_F_ts_true: float,
    counterfactual_F_ts_false: float,
    counterfactual_D_true: float | null,
    counterfactual_D_false: float | null,
    delta_F_true: float,
    delta_F_false: float,
    delta_D_true: float | null,
    delta_D_false: float | null,
    max_abs_delta_D: float | null,
    rank_by_impact_in_topic: int,
    computation_timestamp: ISO8601,
    sensitivity_version: string
  }

COMPUTATION RULES:
1. Compute baseline F_{t,s} = mean(p) over all selected empirical premises in (topic, side) using current p values.
2. Compute baseline D_{r,Frame} = Overall_FOR - Overall_AGAINST for the relevant replicate/frame (§17).
3. Counterfactual A (assumed true): Set this premise's p = 1.0. Recompute F_{t,s} and D_{r,Frame} holding all other premises constant.
4. Counterfactual B (assumed false): Set this premise's p = 0.0. Recompute F_{t,s} and D_{r,Frame} holding all other premises constant.
5. delta_F_true = counterfactual_F_ts_true - baseline_F_ts
   delta_F_false = counterfactual_F_ts_false - baseline_F_ts
   delta_D_true = counterfactual_D_true - baseline_D
   delta_D_false = counterfactual_D_false - baseline_D
6. max_abs_delta_D = max(|delta_D_true|, |delta_D_false|)
7. Rank all premises in the topic-side by max_abs_delta_D. Publish rank.

SCOPE:
- Computed for every empirical premise, not only INSUFFICIENT ones.
- For SUPPORTED/REFUTED premises, the counterfactual shows what would happen if they were flipped (diagnostic only).
- For INSUFFICIENT premises, this is the primary input to §4.12 Decisive Premise Impact Ranking and §17 "what would change this."

PUBLICATION:
- Per-premise InsufficiencySensitivity objects are included in the dossier output.
- Aggregate sensitivity tables are published per topic-side-frame.
- Small-count suppression applies to aggregate tables.

4.12 DECISIVE PREMISE IMPACT RANKING
------------------------------------------------------------

PURPOSE:
Fulfill §17 requirement to publish "top decisive empirical premises currently INSUFFICIENT" and feed the "what would change this" dossier output.

INPUT:
- All FactCheckResults for a topic-side-frame
- All InsufficiencySensitivity records for the same topic-side-frame

OUTPUT:
DecisivePremiseRanking:
  {
    topic_id: string,
    side: enum {FOR, AGAINST},
    frame_id: string | null,
    ranked_premises: list[{
      rank: int,
      premise_id: string,
      current_status: enum {SUPPORTED, REFUTED, INSUFFICIENT},
      current_p: float,
      max_abs_delta_D: float,
      delta_D_true: float,
      delta_D_false: float,
      decisive_evidence_tier: int | null,
      limiting_evidence_tier: int | null,
      insufficiency_reason: string | null,
      human_review_flags: list[string],
      provenance_spans: list[{span_id, post_id, offsets, span_text}],
      operationalization: string,
      citations: list[string]
    }],
    top_k_threshold: int,
    publication_timestamp: ISO8601
  }

RANKING RULES:
1. For every premise, read max_abs_delta_D from its InsufficiencySensitivity record.
2. Sort premises by max_abs_delta_D descending.
3. Publish top-K premises (K is published per snapshot; default 10).
4. Within the top-K, INSUFFICIENT premises are annotated with:
   - What evidence would change them (operationalization)
   - Their current insufficiency_reason
   - Their provenance spans (§7 traceability)
   - Their citations (if any evidence was found but insufficient)
5. SUPPORTED/REFUTED premises may appear in the ranking for diagnostic transparency, but are visually or structurally distinguished from INSUFFICIENT ones.

SELECTION FOR DOSSIER:
- The §17 "what would change this" output includes the top-K INSUFFICIENT premises from DecisivePremiseRanking.
- If a premise has human_review_flags including HIGH_IMPACT_INSUFFICIENT, it is guaranteed inclusion in the dossier even if outside top-K by raw delta_D.
- The dossier also includes the top decisive normative premises with highest judge dispersion (this comes from §14, not the fact-check subsystem).

============================================================
6. INTEGRATION WITH LSD TWO-LOOP SYSTEM
============================================================

6.1 Loop A — Intake Integration
------------------------------------------------------------

The fact-checking system does not operate during Loop A. Loop A produces:
- Moderated posts
- Extracted spans
- Argument Units (AUs) with empirical premises
- Canonicalized premises

At the end of Loop A, the system publishes:
- Snapshot of all canonical empirical premises selected for checking (§11)
- Selection manifest (which premises, why, from which strata)

6.2 Loop B — Adjudication Integration
------------------------------------------------------------

Fact-checking runs as a sub-pipeline within Loop B:

1. RECEIVE: Selected canonical empirical premises (K_E per topic-side, §11.1), each carrying provenance_spans from upstream AU extraction.
2. BATCH: Group by topic-side for parallel processing.
3. PROCESS: Run each premise through §4 pipeline (decomposition → linking → policy → router → normalizer → synthesis).
4. SENSITIVITY: For each completed premise, run §4.11 Insufficiency Sensitivity Engine to compute counterfactual F_{t,s} and D_{r,Frame} values.
5. RANK: Run §4.12 Decisive Premise Impact Ranking per topic-side-frame to produce top-K decisive premises.
6. AGGREGATE: Compute per-topic-side empirical scores (§15A)
   - F_{t,s} = mean(p) over selected empirical premises
   - F_supported_only = mean(p) over SUPPORTED/REFUTED only
   - insufficiency_rate = count(INSUFFICIENT) / count(total)
7. PUBLISH:
   - Per-premise FactCheckResults (including provenance_spans, insufficiency_sensitivity)
   - Per-topic-side DecisivePremiseRanking
   - Per-topic-side aggregate statistics
   - Tier distribution counts
   - Insufficiency sensitivity analysis (§13.1)
8. FEED INTO: §15A (Empirical factuality component) as premise validity inputs.
   Fact-check results are also available to judges for §15C/D context if needed, but they do not replace judge reasoning strength or coverage scores.
   DecisivePremiseRanking feeds directly into §17 "what would change this" dossier output.

6.3 Snapshot Compatibility
------------------------------------------------------------

All fact-check outputs are:
- Immutable after generation
- Hashed and Merkle-rooted (over authoritative_result_hash only)
- Referenced in snapshot replay manifest
- Published with small-count suppression where applicable
- Provenance spans preserved end-to-end for §7 traceability

8. RISK MITIGATION & FAILURE MODES
============================================================

8.1 Connector Failures
- Fallback chain: configurable per connector type in published fallback table
- If all connectors fail for a premise → INSUFFICIENT, reason="connector_failure"
- Circuit breaker pattern for repeated failures
- Connector health published in snapshot diagnostics

8.2 LLM Hallucination in Decomposition
- Validation layer rejects decompositions with novel claims
- Validation layer rejects ClaimExpression trees that are not semantically equivalent to the original premise
- Human spot-check on 5% of decompositions per snapshot
- Publish decomposition overlap rates (§9 stability check)

8.3 Entity Linking Errors
- Confidence threshold at 0.7
- Ambiguity flag triggers INSUFFICIENT rather than wrong link
- Cross-reference linking with multiple APIs

8.4 Evidence Policy Gaps
- Default policy: INSUFFICIENT if no policy matches
- Public petition track for policy additions (§5.3 analog)
- Emergency policy updates via §20.2 protocol

8.5 Temporal Staleness
- Snapshot freeze means evidence is point-in-time
- Publish evidence retrieval timestamps
- Do not attempt "live" updates within snapshot

8.6 Scientific Overgeneralization
- Single studies may support study-specific claims but should not establish broad consensus claims by themselves
- Broad scientific, safety, efficacy, and causal claims require scope-aware policies and cross-verification
- If the retrieved evidence supports a narrower claim than the premise asserts, mark INSUFFICIENT or add SCIENTIFIC_SCOPE_OVERCLAIM

8.7 Tier 3 Promotion Risk
- Tier 3 live search is a discovery mechanism, not a final authority
- Promotion from Tier 3 discovery to Tier 2 evidence requires Source Reputation Registry match + independence checks + policy approval
- If only Tier 3 evidence exists, the automated status remains INSUFFICIENT

8.8 Human Review Queue Saturation
- If too many claims are flagged, publish aggregate flag counts and prioritize review by impact on D, tier conflict, and insufficiency sensitivity
- High-impact unresolved claims feed the "what would change this" dossier output

8.9 Display Summary Consistency Failure
- Consistency checker runs on every display summary
- If FAIL: flag LLM_VALIDATION_FAILURE, add to human review queue, publish machine-generated fallback
- Do not publish inconsistent LLM prose as authoritative

8.10 Logical Structure Misclassification
- Validation layer checks that decomposed ClaimExpression tree is semantically equivalent to the original premise
- If nested logic is flattened or misclassified, flag POLICY_GAP
- Aggregation engine uses the ClaimExpression tree; structural mismatch causes wrong premise-level verdict

8.11 Web Evidence Archiving Failure
- If archive.org permalink generation fails, system MUST store full page content in snapshot storage
- If storage also fails, flag CONNECTOR_FAILURE and mark evidence as Tier 3 with insufficiency risk
- Publish archive success rate per snapshot as diagnostic
- All web evidence MUST have either archive permalink, content hash, or full content stored before inclusion in synthesis

8.12 Provenance Span Loss
- If upstream AU extraction loses span data, flag at intake and do not route to fact-check until resolved
- Validation layer verifies provenance_spans present on every input premise
- Audit record includes input_provenance_spans to detect downstream loss

8.13 Direction Classification Uncertainty
- If direction_confidence < 0.7, direction is set to UNCLEAR
- An UNCLEAR item cannot alone determine SUPPORTED or REFUTED
- For structured sources, deterministic comparison is preferred over LLM classification

8.14 Scope Mismatch
- If evidence verdict_scope is narrower than the premise/subclaim scope, the synthesis engine marks INSUFFICIENT with reason "evidence_scope_narrower_than_claim"
- This prevents overgeneralized verdicts from geographically or temporally limited evidence

APPENDIX A: Claim Decomposition Examples
------------------------------------------------------------

EXAMPLE 1 — Compound Conjunction:
Input: "The 2023 tax cut increased GDP growth and reduced unemployment."
Decomposition:
  claim_expression:
    node_type: AND
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"
      - node_type: ATOMIC, subclaim_id: "sc-2"
  1. "The 2023 tax cut was enacted" [TEMPORAL/LEGAL]
  2. "GDP growth increased after the 2023 tax cut" [CAUSAL/NUMERIC]
  3. "Unemployment reduced after the 2023 tax cut" [CAUSAL/NUMERIC]

EXAMPLE 2 — Disjunction:
Input: "Either the policy reduced unemployment or it increased wages."
Decomposition:
  claim_expression:
    node_type: OR
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"
      - node_type: ATOMIC, subclaim_id: "sc-2"
  1. "The policy reduced unemployment" [CAUSAL/NUMERIC]
  2. "The policy increased wages" [CAUSAL/NUMERIC]

EXAMPLE 3 — Conditional:
Input: "If the tax cut was enacted, then small business investment rose."
Decomposition:
  claim_expression:
    node_type: IF_THEN
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"  // antecedent A
      - node_type: ATOMIC, subclaim_id: "sc-2"  // consequent B
  1. "The tax cut was enacted" [TEMPORAL/LEGAL]
  2. "Small business investment rose" [NUMERIC_STATISTICAL]

EXAMPLE 4 — Negation:
Input: "The policy did not reduce unemployment."
Decomposition:
  claim_expression:
    node_type: NOT
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"
  1. "The policy reduced unemployment" [CAUSAL/NUMERIC]

EXAMPLE 5 — Nested Logic:
Input: "The tax cut increased GDP and either reduced unemployment or increased wages."
Decomposition:
  claim_expression:
    node_type: AND
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"
      - node_type: OR
        children:
          - node_type: ATOMIC, subclaim_id: "sc-2"
          - node_type: ATOMIC, subclaim_id: "sc-3"

EXAMPLE 6 — Comparison:
Input: "The policy increased wages more than it reduced unemployment."
Decomposition:
  claim_expression:
    node_type: COMPARISON
    operator: "more_than"
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"
      - node_type: ATOMIC, subclaim_id: "sc-2"

EXAMPLE 7 — Quantifier:
Input: "Most small businesses benefited from the tax cut."
Decomposition:
  claim_expression:
    node_type: QUANTIFIER
    quantifier: "MOST"
    children:
      - node_type: ATOMIC, subclaim_id: "sc-1"

EXAMPLE 8 — Ambiguous/Normative:
Input: "This policy is unfair to small businesses."
Classification: NORMATIVE → Route to §14, not fact-check

EXAMPLE 9 — Predictive:
Input: "AI will destroy 40% of jobs by 2030."
Classification: PREDICTIVE → INSUFFICIENT, reason="predictive_claim_not_checkable"
  (Can decompose to: "Studies project X job loss" [SCIENTIFIC] which is checkable)

APPENDIX B: Evidence Tier Assignment Rules
------------------------------------------------------------

TIER 1 (Primary/Official):
- Government statistical releases
- Laws and regulations (official texts)
- Court decisions (official records)
- Peer-reviewed primary research for study-specific claims
- Systematic reviews, meta-analyses, or major scientific body statements for broad consensus/generalized claims
- Official organizational reports (UN, WHO, etc.)

TIER 2 (Reputable Secondary Synthesis):
- Wikipedia (with cited sources)
- Major news organizations (Reuters, AP, BBC, etc.)
- Academic review papers
- Reputable think tank reports (with methodology)
- Wikidata statements with reputable secondary references
- Primary/official sources discovered through Wikidata references should be attributed to the underlying primary source, with retrieval_path = WIKIDATA_REFERENCE

TIER 3 (Limited/Uncertain):
- Unreferenced web content
- Opinion pieces
- Preliminary reports
- Wikidata statements without references
- Single-source news blogs

APPENDIX D: Gold Test Set Categories
------------------------------------------------------------

Minimum initial test coverage (suggest 5-7 examples per category):
- SUPPORTED structured entity relation
- REFUTED structured entity relation
- INSUFFICIENT due to Wikidata absence
- Temporal claim with start/end qualifiers
- Numeric statistical claim with official source
- Scoped numeric claim requiring date/geography/rounding tolerance
- Compound CONJUNCTION with one false subclaim
- Compound CONJUNCTION with one insufficient subclaim
- Compound DISJUNCTION with mixed subclaim statuses
- Compound CONDITIONAL with insufficient antecedent
- Compound CONDITIONAL with refuted antecedent (must yield INSUFFICIENT, not SUPPORTED)
- Negation claim (NOT node)
- Nested logic (AND containing OR)
- Comparison claim (more than / less than)
- Quantifier claim (MOST / AT_LEAST_N)
- Ambiguous entity claim
- Current-event claim requiring two independent sources
- Scientific claim scoped to one study
- Scientific broad-consensus claim requiring synthesis
- Legal/regulatory claim requiring official source
- Predictive claim marked INSUFFICIENT unless reframed
- Normative claim routed out of empirical fact-checking
- Connector failure case
- Contradictory Tier 1 evidence case (resolved by authority ranking)
- Contradictory Tier 1 evidence case (unresolved, INSUFFICIENT)
- Tier 3 promotion case (source in reputation registry)
- Tier 3 non-promotion case (source not in registry)
- Provenance span traceability case
- Insufficiency sensitivity / decisive ranking case
- Scope mismatch case (evidence narrower than claim)
- Direction classification uncertainty case (low confidence → UNCLEAR)

APPENDIX E: Verdict Scope Fields
------------------------------------------------------------

verdict_scope records the exact interpretation under which the verdict applies:
- temporal_scope: date, date range, event time, or publication time
- geographic_scope: country, region, jurisdiction, locality, or global scope
- population_scope: affected population, sample, organization, demographic group
- measurement_definition: metric definition, unit, denominator, inclusion/exclusion criteria
- source_basis: source family used, e.g., official census, court record, peer-reviewed paper
- rounding_tolerance: numeric tolerance accepted for approximations

APPENDIX F: Glossary
------------------------------------------------------------

AU — Argument Unit (§8)
Frame — Normative evaluation lens (§5)
K_E — Compute budget: empirical premises to check (§11.1)
LSD — Large-Scale Debate system
p — Probability of truth (§13)
Q_{t,s,Frame} — Component combination score (§16)
Span — Traceable text segment (§7)
Artifact replay — Reconstructing a snapshot from frozen records and hashes
Computational rerun — Re-executing connectors/model steps from a replay manifest
HumanReviewFlag — Diagnostic flag that routes a fact-check case to governance/human review without changing p by itself
Verdict scope — Metadata describing the exact date, geography, measurement definition, source basis, and tolerance under which a verdict applies
SynthesisLogic — Structured, rule-generated proof trace that is the authoritative rationale for a fact-check result
DisplaySummary — Optional, non-authoritative human-readable prose generated from SynthesisLogic
Source Reputation Registry — Allowlist of domains/publications that qualify for Tier 2 promotion from Tier 3 discovery
InsufficiencySensitivity — Counterfactual computation showing how F_{t,s} and D_{r,Frame} change if a premise were SUPPORTED or REFUTED
DecisivePremiseRanking — Ranked list of premises by their potential impact on the overall debate verdict
ClaimExpression — Recursive tree structure representing the logical form of a decomposed premise (ATOMIC, AND, OR, NOT, IF_THEN, COMPARISON, QUANTIFIER)
