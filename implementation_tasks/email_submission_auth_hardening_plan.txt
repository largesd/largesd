TITLE
  Require login for proposal and all posting flows, including GitHub/email mode

STATUS
  Implementation plan only. No code changes are included in this task file.

PROBLEM
  The product policy is:
    - Anonymous users may view debate pages, snapshots, dossiers, topics, and public results.
    - Login is required before creating any write-intent artifact:
        - debate proposals
        - direct API posts
        - GitHub/email-mode posts

  The current code mostly enforces this, but there is one important gap:
    - Direct API posting is protected by @login_required.
    - Debate proposal submission is protected by @login_required and the propose page redirects
      unauthenticated users to login.
    - GitHub/email posting can generate a local mailto: submission without login when
      DataBridge is configured.
    - The email processor can also receive messages directly at the mailbox, so a frontend-only
      login gate would reduce accidental/drive-by abuse but would not fully enforce the policy.

GOALS
  1. Preserve anonymous viewing.
  2. Require login before a user can generate or submit an email-mode post.
  3. Enforce authentication again in the email ingestion daemon so users cannot bypass the UI by
     emailing the processor directly.
  4. Keep the system identity-blind on public debate surfaces. Authentication is for abuse
     reduction and auditability, not for public ranking, profiles, or reputation.
  5. Provide clear user-facing behavior:
       - unauthenticated visitors can read
       - clicking "Post Argument" or "Propose" redirects to login
       - after login, the user returns to the intended page
       - email submissions from unregistered/invalid users get a rejection acknowledgment

NON-GOALS
  - Do not require login to view public debates, topics, snapshots, verdicts, dossiers, evidence
    pages, governance read pages, or GitHub-published consolidated results.
  - Do not introduce public author identity on debate posts.
  - Do not make admin access broader.
  - Do not replace the email ingestion pipeline with direct GitHub writes.

        - init redirects unauthenticated users to login.

        - submitPost generates a local mailto link through DataBridge.buildMailtoLink.
    - frontend/static/js/data_bridge.js

RECOMMENDED SOLUTION
  Implement this in two layers:

  Layer 1: UI gate
    - new_debate.html/new_debate.js should require login before the posting UI is usable,
      including GitHub/email mode.
    - If a static/GitHub-only deployment has no live auth backend, email-mode posting must be
      disabled with a clear login/setup message. It should not silently generate unsigned mail.

  Layer 2: ingestion enforcement
    - Email submissions should include a signed auth token generated by the backend after login.
    - The email processor must verify that token before accepting the message.
    - The verified user_id should be passed into DebateEngineV2.submit_post so the stored post has
      a user_id internally, while public surfaces remain identity-blind.

  This avoids relying on From headers alone. From headers can be spoofed, forwarded, or modified
  by clients. Registered-sender checks can be useful as a secondary defense, but the primary
  enforcement should be a signed, expiring submission token.

PHASE 3 - ADD AN AUTHENTICATED EMAIL DRAFT ENDPOINT
  Files:
    - backend/routes/debate_bp.py or a new backend/routes/email_submission_bp.py
    - backend/routes/__init__.py if a new blueprint is created
    - backend/email_submission_parser.py
    - backend/utils/decorators.py or a new backend/email_submission_auth.py helper

  Preferred endpoint:
    - POST /api/debate/email-submission-draft
    - Decorator: @login_required

  Request JSON:
    {
      "debate_id": "debate_...",
      "side": "FOR",
      "topic_id": "t1",
      "facts": "...",
      "inference": "...",
      "counter_arguments": "..."
    }

  Response JSON:
    {
      "version": "BDA Submission v3",
      "submission_id": "...",
      "submitted_at": "...",
      "expires_at": "...",
      "body": "...",
      "subject": "BDA Submission - FOR - debate_...",
      "mailto": "mailto:..."
    }

  Validation:
    - Require login.
    - Validate debate exists.
    - Validate public/private debate access consistently with get_debate_by_id.
    - Validate side, topic_id, facts, inference, and counter_arguments using existing validators
      where possible.
    - Sanitize facts/inference/counter_arguments before placing them in the email body.
    - Use PROCESSOR_DEST_EMAIL, or a dedicated configured destination, for the mailto address.
      If no destination is configured, return a 400 with a clear code such as EMAIL_DEST_MISSING.

  Token design:
    - Use PyJWT, already used for normal auth tokens.
    - Create a token with a distinct type:
        {
          "type": "email_submission",
          "user_id": g.user["user_id"],
          "email": g.user["email"],
          "debate_id": debate_id,
          "submission_id": submission_id,
          "exp": now + EMAIL_SUBMISSION_TOKEN_TTL_MINUTES,
          "iat": now
        }
    - Sign with SECRET_KEY using HS256, or use a dedicated EMAIL_SUBMISSION_SECRET if configured.
    - Default TTL: 24 hours. Make it configurable.

  Email body format:
    - Introduce "BDA Submission v3".
    - Include a required Auth-Token field.
    - Keep the existing human-readable fields for audit/debugging.

    Example:
      BDA Submission v3
      Debate-ID: debate_abc
      Resolution: Resolved: ...
      Submission-ID: 123e4567-e89b-12d3-a456-426614174000
      Submitted-At: 2026-05-06T12:00:00Z
      Expires-At: 2026-05-07T12:00:00Z
      Submitter-Email: user@example.com
      Position: FOR
      Topic-Area: t1
      Auth-Token: <jwt>

      Facts:
      ...

      Inference:
      ...

      Counter-Arguments:
      ...

  Parser changes:
    - Add HEADER_V3_RE.
    - Add auth_token: str | None to EmailSubmission.
    - Add submitter_user_id: str | None only if useful; the processor can also derive user_id
      after token verification.
    - Add "Auth-Token" to REQUIRED_FIELDS for v3 only.
    - Do not add Auth-Token to v1/v2 REQUIRED_FIELDS if old tests still cover v1/v2 parsing.
    - Add build_email_body_v3(...) or extend build_email_body(..., auth_token=None,
      expires_at=None, version="v3") carefully.
    - Keep v1/v2 parsing for compatibility, but enforcement will reject them when auth is required.

  Acceptance:
    - Logged-out request to POST /api/debate/email-submission-draft returns 401.
    - Logged-in request returns a v3 body containing Auth-Token.
    - Returned mailto uses the configured destination email.
    - The token validates with the backend secret and contains user_id, email, debate_id,
      submission_id, exp, iat, and type=email_submission.

PHASE 5 - VERIFY EMAIL SUBMISSION TOKENS IN THE PROCESSOR
  Files:
    - backend/email_processor.py
    - backend/email_submission_parser.py
    - backend/email_submission_auth.py, recommended new helper
    - backend/debate_engine_v2.py only if needed

  New config:
    - EMAIL_SUBMISSION_REQUIRE_AUTH=true by default
    - EMAIL_SUBMISSION_TOKEN_TTL_MINUTES=1440
    - EMAIL_SUBMISSION_AUTH_ALLOW_LEGACY=false by default
    - EMAIL_SUBMISSION_SECRET optional, fallback SECRET_KEY

  Verification flow in _process_message(), after parse and before deduplication:
    1. If EMAIL_SUBMISSION_REQUIRE_AUTH is true:
         - Require submission.auth_token.
         - Decode JWT.
         - Require type == "email_submission".
         - Require debate_id in token == submission.debate_id.
         - Require submission_id in token == submission.submission_id.
         - Require email in token matches normalized submitter_email, unless explicitly allowing
           aliases later.
         - Require user_id exists in the database.
         - Require user is active. If is_verified is meaningful in this app, decide whether
           verified email is required and document it.

    2. If token verification fails:
         - Do not submit the post.
         - Send a rejection ack:
             "Email submission rejected: authentication token is missing, expired, or invalid.
              Please log in and regenerate the email submission."
         - Do not mark as processed unless the existing processor policy marks rejected mail as
           seen. Prefer marking invalid structured submissions as seen after ack to avoid repeated
           rejection loops, but decide consistently with current ops expectations.

    3. Pass verified user_id into submit_post:
         post = self.debate_engine.submit_post(
           ...,
           user_id=verified_user["user_id"],
           submission_id=submission.submission_id,
         )

  Optional secondary defense:
    - Keep SENDER_WHITELIST support.
    - Registered sender check can be used as a fallback only if explicitly configured:
        EMAIL_SUBMISSION_AUTH_MODE=registered_sender
      but the recommended default is signed_token.

  Acceptance:
    - v3 email with valid token is processed and saved with user_id.
    - v3 email with missing token is rejected.
    - v3 email with expired token is rejected.
    - v3 email whose token debate_id/submission_id/email does not match the body or sender is
      rejected.
    - v1/v2 email is rejected when EMAIL_SUBMISSION_REQUIRE_AUTH=true.

RISK NOTES
  - Frontend-only login enforcement is insufficient because users can email the processor directly.
  - Sender email alone is weak authentication. Use it as a consistency check, not the primary
    proof of authentication.
  - Mail clients may wrap long JWT lines. The parser should trim whitespace and tolerate line
    wrapping only if implemented safely. Simpler option: keep Auth-Token as one line and reject
    wrapped tokens with a clear ack.
  - Tokens in email bodies are bearer credentials for one submission. Keep the TTL short enough to
    limit replay risk. Deduplication by submission_id already helps prevent repeated processing.
  - If a user forwards a generated email before sending, the From address/token email mismatch
    should reject the forwarded submission.
  - Do not expose user_id or email on public result surfaces.

DEFINITION OF DONE
  - Anonymous users can still view public debate information.
  - Anonymous users cannot submit debate proposals.
  - Anonymous users cannot submit direct API posts.
  - Anonymous users cannot generate email-mode submissions.
  - Direct emails without a valid signed auth token are rejected by the processor.
  - Valid signed email submissions are processed and stored with internal user_id.
  - Tests cover API auth, parser behavior, token verification, processor acceptance/rejection,
    and the new frontend flow.
