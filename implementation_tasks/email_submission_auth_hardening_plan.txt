TITLE
  Require login for proposal and all posting flows, including GitHub/email mode

STATUS
  Implementation plan only. No code changes are included in this task file.

PROBLEM
  The product policy is:
    - Anonymous users may view debate pages, snapshots, dossiers, topics, and public results.
    - Login is required before creating any write-intent artifact:
        - debate proposals
        - direct API posts
        - GitHub/email-mode posts

  The current code mostly enforces this, but there is one important gap:
    - Direct API posting is protected by @login_required.
    - Debate proposal submission is protected by @login_required and the propose page redirects
      unauthenticated users to login.
    - GitHub/email posting can generate a local mailto: submission without login when
      DataBridge is configured.
    - The email processor can also receive messages directly at the mailbox, so a frontend-only
      login gate would reduce accidental/drive-by abuse but would not fully enforce the policy.

GOALS
  1. Preserve anonymous viewing.
  2. Require login before a user can generate or submit an email-mode post.
  3. Enforce authentication again in the email ingestion daemon so users cannot bypass the UI by
     emailing the processor directly.
  4. Keep the system identity-blind on public debate surfaces. Authentication is for abuse
     reduction and auditability, not for public ranking, profiles, or reputation.
  5. Provide clear user-facing behavior:
       - unauthenticated visitors can read
       - clicking "Post Argument" or "Propose" redirects to login
       - after login, the user returns to the intended page
       - ack-safe invalid email submissions get a rejection acknowledgment, while untrusted or
         unparseable mail is dropped/logged without outbound backscatter

NON-GOALS
  - Do not require login to view public debates, topics, snapshots, verdicts, dossiers, evidence
    pages, governance read pages, or GitHub-published consolidated results.
  - Do not introduce public author identity on debate posts.
  - Do not make admin access broader.
  - Do not replace the email ingestion pipeline with direct GitHub writes.

PREREQUISITES
  Before beginning implementation, verify the following in the target environment:
    1. PyJWT is installed and available to both the web backend and the email processor.
    2. The posts / submissions database schema already supports a user_id column, or a migration
       has been written to add it. The plan assumes DebateEngineV2.submit_post(...) accepts
       user_id, but the underlying table must also store it.
    3. PROCESSOR_DEST_EMAIL is configured in the target environment, or email-mode posting will
       be disabled with a clear user-facing message.
    4. The existing SENDER_WHITELIST config (if used) is documented and its interaction with the
       new token system is understood.

RECOMMENDED SOLUTION
  Implement this in two layers:

  Layer 1: UI gate
    - new_debate.html/new_debate.js should require login before the posting UI is usable,
      including GitHub/email mode.
    - If a static/GitHub-only deployment has no live auth backend, email-mode posting must be
      disabled with a clear login/setup message. It should not silently generate unsigned mail.

  Layer 2: ingestion enforcement
    - Email submissions should include a signed auth token generated by the backend after login.
    - The email processor must verify that token before accepting the message.
    - The verified user_id should be passed into DebateEngineV2.submit_post so the stored post has
      a user_id internally, while public surfaces remain identity-blind.

  This avoids relying on From headers alone. From headers can be spoofed, forwarded, or modified
  by clients. Registered-sender checks can be useful as a secondary defense, but the primary
  enforcement should be a signed, expiring submission token.

PHASE 3 - ADD AN AUTHENTICATED EMAIL DRAFT ENDPOINT
  Files:
    - backend/routes/debate_bp.py or a new backend/routes/email_submission_bp.py
    - backend/routes/__init__.py if a new blueprint is created
    - backend/email_submission_parser.py
    - backend/email_submission_auth.py, required shared helper for token and payload hashing
    - backend/email_submission_payload.py, optional shared helper if payload canonicalization is
      large enough to keep separate from token helpers

  Preferred endpoint:
    - POST /api/debate/email-submission-draft
    - Decorator: @login_required

  Request JSON:
    {
      "debate_id": "debate_...",
      "side": "FOR",
      "topic_id": "t1",
      "facts": "...",
      "inference": "...",
      "counter_arguments": "..."
    }

  Response JSON:
    {
      "version": "BDA Submission v3",
      "submission_id": "...",
      "submitted_at": "...",
      "expires_at": "...",
      "payload_hash_alg": "sha256",
      "payload_hash": "...",
      "body": "...",
      "subject": "BDA Submission - FOR - debate_...",
      "mailto": "mailto:..."
    }

  Validation:
    - Require login.
    - Validate debate exists.
      If it does not exist in the live backend, return 404 or 409 with
      code=DEBATE_NOT_AVAILABLE_FOR_POSTING.
    - Validate public/private debate access consistently with get_debate_by_id.
    - Validate side, topic_id, facts, inference, and counter_arguments using existing validators
      where possible.
    - Sanitize facts/inference/counter_arguments before placing them in the email body.
    - Use the sanitized values, not the raw request values, for the generated body, preview body,
      payload hash, and later processor comparison.
    - Use PROCESSOR_DEST_EMAIL, or a dedicated configured destination, for the mailto address.
      If no destination is configured, return a 400 with a clear code such as EMAIL_DEST_MISSING.

  Rate limiting:
    - Add per-user rate limiting on the draft endpoint to prevent token generation abuse.
    - Suggested policy: max 5 drafts per user per hour, tracked in memory (e.g., Flask-Limiter)
      or in a lightweight cache (e.g., Redis if available, otherwise an in-process dict with TTL).
    - Return 429 with Retry-After header if the limit is exceeded.
    - Rate limit should be keyed by authenticated user_id, not by IP, to avoid penalizing shared
      networks.

  Token design:
    - Use PyJWT, already used for normal auth tokens.
    - Create a canonical submission payload after validation and sanitization. Recommended helper:
        canonical_email_submission_payload({
          "debate_id": debate_id,
          "side": side,
          "topic_id": topic_id,
          "facts": sanitized_facts,
          "inference": sanitized_inference,
          "counter_arguments": sanitized_counter_arguments or ""
        })
      The canonical form must be stable JSON:
        - sort keys
        - compact separators
        - UTF-8 encoded
        - no timestamps, subject, resolution text, token, or mailto address
      Do not normalize internal user text beyond the same trim/sanitize rules used before writing
      the email body. Editing the email body should change the hash and cause rejection.
    - Compute:
        payload_hash = base64url_no_padding(sha256(canonical_payload_bytes))
        payload_hash_alg = "sha256"
    - Create a token with a distinct type and payload binding:
        {
          "type": "email_submission",
          "user_id": g.user["user_id"],
          "email": g.user["email"],
          "debate_id": debate_id,
          "submission_id": submission_id,
          "side": side,
          "topic_id": topic_id,
          "payload_hash": payload_hash,
          "payload_hash_alg": "sha256",
          "exp": now + EMAIL_SUBMISSION_TOKEN_TTL_MINUTES,
          "iat": now
        }
    - Sign with SECRET_KEY using HS256, or use a dedicated EMAIL_SUBMISSION_SECRET if configured.
    - Default TTL: 24 hours. Make it configurable.
    - Token rotation / secret rotation note: If SECRET_KEY is rotated, all previously issued
      email submission tokens immediately become invalid. This is acceptable because the TTL is
      short (24h by default), but operators should be aware that rotating the main secret will
      invalidate in-flight draft emails. Document this in ops runbooks.

  Shared auth helper requirements:
    - Implement backend/email_submission_auth.py as a Flask-independent module. It must not import
      flask.current_app, flask.g, flask.request, or decorators from backend/utils/decorators.py.
      The email processor runs outside an app context, so hidden Flask dependencies will fail in
      production even if unit tests pass inside a test app.
    - Recommended API:
        @dataclass(frozen=True)
        class EmailSubmissionAuthConfig:
          secret: str
          algorithm: str = "HS256"
          ttl_minutes: int = 1440
          require_verified_email: bool = False

        def compute_payload_hash(payload: dict[str, str]) -> str: ...
        def generate_email_submission_token(config, claims, now=None) -> str: ...
        def decode_email_submission_token(config, token, now=None) -> dict: ...
        def verify_email_submission_claims(config, token, submission, sender_email, user_lookup) -> VerifiedEmailSubmission: ...
    - The web endpoint should build EmailSubmissionAuthConfig from current_app.config or env and
      pass plain values into the helper.
    - The email processor should build the same config from EmailProcessorConfig/env and pass plain
      values into the helper.
    - Add an acceptance test that creates a draft token through the Flask endpoint, then verifies
      that token by calling the helper in a standalone context with no Flask application active.
    - Keep normal session JWT helpers in backend/utils/decorators.py unchanged. Email submission
      tokens are a different credential type and should not share decorator-level assumptions.

  Email body format:
    - Introduce "BDA Submission v3".
    - Include a required Auth-Token field.
    - Include required Payload-Hash and Payload-Hash-Alg fields for audit/debugging. The token's
      payload_hash claim is authoritative; the body Payload-Hash must match it.
    - Keep the existing human-readable fields for audit/debugging.

    Example:
      BDA Submission v3
      Debate-ID: debate_abc
      Resolution: Resolved: ...
      Submission-ID: 123e4567-e89b-12d3-a456-426614174000
      Submitted-At: 2026-05-06T12:00:00Z
      Expires-At: 2026-05-07T12:00:00Z
      Submitter-Email: user@example.com
      Position: FOR
      Topic-Area: t1
      Payload-Hash-Alg: sha256
      Payload-Hash: <base64url sha256 hash>
      Auth-Token: <jwt>

      Facts:
      ...

      Inference:
      ...

      Counter-Arguments:
      ...

  Parser changes:
    - Add HEADER_V3_RE.
    - Add auth_token: str | None to EmailSubmission.
    - Add payload_hash: str | None and payload_hash_alg: str | None to EmailSubmission.
    - Add submitter_user_id: str | None only if useful; the processor can also derive user_id
      after token verification.
    - Add "Auth-Token", "Payload-Hash", and "Payload-Hash-Alg" to REQUIRED_FIELDS for v3 only.
    - Do not add these v3 fields to v1/v2 REQUIRED_FIELDS if old tests still cover v1/v2 parsing.
    - Add build_email_body_v3(...) or extend build_email_body(..., auth_token=None,
      expires_at=None, payload_hash=None, payload_hash_alg="sha256", version="v3") carefully.
    - Keep v1/v2 parsing for compatibility, but enforcement will reject them when auth is required.

  Handling Auth-Token line wrapping in email clients:
    - JWTs are often longer than 80 characters and may be wrapped by mail clients.
    - Implement a safe continuation rule in the parser:
        1. When the parser encounters a header line matching "Auth-Token:", capture the value
           on that line.
        2. Continue reading subsequent lines. If the next line starts with whitespace
           (space or tab), treat it as a continuation of the Auth-Token value and append it
           (stripping the leading whitespace).
        3. Stop when a non-whitespace-starting line is encountered.
        4. Strip all internal whitespace from the concatenated token before JWT decoding.
    - Reject tokens that still fail validation after unwrapping, with a clear rejection reason.
    - Document in user-facing guides that the Auth-Token should not be manually edited.

  Acceptance:
    - Logged-out request to POST /api/debate/email-submission-draft returns 401.
    - Logged-in request returns a v3 body containing Auth-Token.
    - Returned mailto uses the configured destination email.
    - The token validates with the backend secret and contains user_id, email, debate_id,
      submission_id, side, topic_id, payload_hash, payload_hash_alg, exp, iat, and
      type=email_submission.
    - The body Payload-Hash matches the token payload_hash claim.
    - Recomputing the canonical hash from the response body's sanitized fields matches
      payload_hash.
    - Rate limit is enforced: sixth request within one hour returns 429.
    - Parser correctly unwraps a multi-line Auth-Token.

PHASE 5 - VERIFY EMAIL SUBMISSION TOKENS IN THE PROCESSOR
  Files:
    - backend/email_processor.py
    - backend/email_submission_parser.py
    - backend/email_submission_auth.py, required shared helper
    - backend/debate_engine_v2.py only if needed

  New config:
    - EMAIL_SUBMISSION_REQUIRE_AUTH=true by default
    - EMAIL_SUBMISSION_TOKEN_TTL_MINUTES=1440
    - EMAIL_SUBMISSION_AUTH_ALLOW_LEGACY=false by default
    - EMAIL_SUBMISSION_SECRET optional, fallback SECRET_KEY
    - EMAIL_SUBMISSION_ACK_UNSIGNED_REJECTIONS=false by default
      This controls whether missing-token or unsigned legacy rejections get outbound acks. The
      recommended public setting is false to reduce backscatter from spoofed From addresses.

  Parse classification in _process_message(), after body extraction and before verification:
    1. Replace the current "ack every EmailSubmissionError" behavior with explicit parse
       classification. Recommended implementation:
         - Add an EmailSubmissionParseResult dataclass, or extend EmailSubmissionError with
           machine-readable fields:
             version_hint: "v1" | "v2" | "v3" | "unknown"
             reason_code: "unrecognized" | "legacy_disabled" | "missing_token" |
                          "missing_required_field" | "malformed_header" | "invalid_body"
             structured: bool
             ack_safe: bool
         - Keep parse_body(...) available for unit tests if useful, but add a processor-facing
           parse_for_processor(...) wrapper that never forces the processor to infer whether an
           exception came from spam, legacy mail, or malformed v3.

    2. Parse-result behavior:
         - unknown/unrecognized mail:
             Log and drop without outbound ack. If MARK_PROCESSED=true, mark the message Seen so
             the daemon does not loop on spam forever.
         - v1/v2 while EMAIL_SUBMISSION_REQUIRE_AUTH=true and legacy is not allowed:
             Reject with reason legacy_disabled. Do not submit. Do not send an outbound ack unless
             EMAIL_SUBMISSION_ACK_UNSIGNED_REJECTIONS=true.
         - v3 missing Auth-Token or required payload-hash headers:
             Reject with reason missing_token or missing_required_field. Do not submit. Do not send
             an outbound ack unless EMAIL_SUBMISSION_ACK_UNSIGNED_REJECTIONS=true.
         - v3 with Auth-Token present:
             Continue to token verification. Whether a rejection ack is sent depends on the
             ack-safe rules below.

  Verification flow in _process_message(), after parse classification and before deduplication:
    1. If EMAIL_SUBMISSION_REQUIRE_AUTH is true:
         - Require submission.auth_token.
         - Decode JWT.
         - Require type == "email_submission".
         - Require debate_id in token == submission.debate_id.
         - Require submission_id in token == submission.submission_id.
         - Require side in token == submission.side.
         - Require topic_id in token == submission.topic_id.
         - Require email in token matches normalized submitter_email, unless explicitly allowing
           aliases later.
         - Require user_id exists in the database.
         - Require user is active. If is_verified is meaningful in this app, decide whether
           verified email is required and document it.

    2. Cross-debate token reuse prevention:
         - Reject any submission where the token's debate_id does not match the Debate-ID header
           in the email body. This prevents an attacker from taking a valid token for Debate A
           and replaying it against Debate B.
         - Similarly, reject if the token's submission_id does not match the Submission-ID header.

    3. Payload edit prevention:
         - Validate parsed side/topic_id/facts/inference/counter_arguments with the same validators
           used by the direct API where possible.
         - Sanitize facts/inference/counter_arguments with the same sanitizer used by the direct API
           and the draft endpoint.
         - Recompute canonical_email_submission_payload(...) from the sanitized parsed fields.
         - Recompute payload_hash and require:
             token.payload_hash == recomputed_payload_hash
             token.payload_hash_alg == "sha256"
             submission.payload_hash == token.payload_hash
             submission.payload_hash_alg == token.payload_hash_alg
         - If any comparison fails, reject with reason payload_mismatch.
         - Use the sanitized parsed fields, not the raw email body text, when calling submit_post.

    4. Token forwarding attack prevention:
         - Normalize the From header email address (lowercase, strip display name).
         - Compare it to the email claim inside the JWT.
         - If they do not match, reject the submission. This prevents a user from forwarding
           their draft email to another person; the forwarded email will have a different From
           address and fail verification.

    5. If token verification fails:
         - Do not submit the post.
         - Send a rejection ack only when ack_safe=true:
             - expired token with a valid signature
             - valid signature but rejected for wrong type, debate mismatch, submission mismatch,
               side/topic mismatch, payload mismatch, email mismatch, unknown user, inactive user,
               or verified-email policy
             - optionally missing-token/legacy rejects only when
               EMAIL_SUBMISSION_ACK_UNSIGNED_REJECTIONS=true
         - Do not send an ack for unrecognized mail, malformed JWTs, invalid signatures, or any
           message where the processor cannot establish that it is replying to a plausible sender.
         - Rejection ack copy:
             "Email submission rejected: authentication token is missing, expired, invalid, or
              does not match the submitted content.
              Please log in and regenerate the email submission."
         - Completely unparseable emails (not v1/v2/v3) should be silently dropped or logged
           without sending an outbound reply.
         - Mark invalid structured submissions as processed (seen) after sending the ack to
           avoid repeated rejection loops. Use a dedicated "rejected" state or flag if the
           existing schema supports it; otherwise, mark as processed with a rejection note.
         - If no ack is sent, still mark rejected structured submissions Seen when
           MARK_PROCESSED=true so the daemon does not retry the same invalid message forever.

    6. Pass verified user_id and sanitized parsed fields into submit_post:
         post = self.debate_engine.submit_post(
           ...,
           user_id=verified_user["user_id"],
           submission_id=submission.submission_id,
         )

  Optional secondary defense:
    - Keep SENDER_WHITELIST support.
    - Registered sender check can be used as a fallback only if explicitly configured:
        EMAIL_SUBMISSION_AUTH_MODE=registered_sender
      but the recommended default is signed_token.

  Observability:
    - Add structured logging and/or metrics for every email submission decision:
        - email_submissions_total{version="v3", decision="accepted"}

  Acceptance:
    - v3 email with valid token is processed and saved with user_id.
    - v3 email with missing token is rejected.
    - v3 email with expired token is rejected.
    - v3 email whose token debate_id/submission_id/email does not match the body or sender is
      rejected.
    - v3 email whose side/topic/body content was edited after draft generation is rejected with
      payload_mismatch and does not call submit_post.
    - v1/v2 email is rejected when EMAIL_SUBMISSION_REQUIRE_AUTH=true.
    - Forwarded email (valid token, different From header) is rejected.
    - Cross-debate token reuse is rejected.
    - Unparseable spam is silently dropped (no outbound rejection email).
    - Missing-token v3 mail is rejected without an outbound ack by default; enabling
      EMAIL_SUBMISSION_ACK_UNSIGNED_REJECTIONS changes only ack behavior, not acceptance.
    - Metrics/logs capture every decision with sufficient detail.

RISK NOTES
  - Frontend-only login enforcement is insufficient because users can email the processor directly.
  - Sender email alone is weak authentication. Use it as a consistency check, not the primary
    proof of authentication.
  - A signed token that is not bound to the submitted side/topic/content only proves who generated
    a draft, not what they generated. Payload hashing is required so edited email bodies are
    rejected instead of bypassing draft-endpoint validation.
  - Mail clients may wrap long JWT lines. The parser should trim whitespace and tolerate line
    wrapping only if implemented safely. Simpler option: keep Auth-Token as one line and reject
    wrapped tokens with a clear ack. (See Phase 3 parser guidance for the recommended approach.)
  - Tokens in email bodies are bearer credentials for one submission. Keep the TTL short enough to
    limit replay risk. Deduplication by submission_id already helps prevent repeated processing.
  - If a user forwards a generated email before sending, the From address/token email mismatch
    should reject the forwarded submission.
  - The email processor is not a Flask request handler. Shared token helpers must accept explicit
    config and must be testable without a Flask app context.
  - Do not expose user_id or email on public result surfaces.
  - Backscatter risk: sending outbound rejection emails to spoofed senders can harm domain
    reputation. Only send rejection acks for ack-safe structured v3 failures; drop unparseable
    mail silently.
  - Rate limit the draft endpoint to prevent token generation abuse.

DEFINITION OF DONE
  - Anonymous users can still view public debate information.
  - Anonymous users cannot submit debate proposals.
  - Anonymous users cannot submit direct API posts.
  - Anonymous users cannot generate email-mode submissions.
  - Direct emails without a valid signed auth token are rejected by the processor.
  - Emails whose content no longer matches the signed payload hash are rejected by the processor.
  - Valid signed email submissions are processed and stored with internal user_id.
  - A token generated by the web draft endpoint verifies in the standalone email processor helper.
  - Logged-out new_debate.html redirects to login.html, not setup.html, even when DataBridge is
    unconfigured.
  - Cache-only debates cannot generate signed email drafts unless the live backend validates the
    debate_id.
  - Tests cover API auth, parser behavior, token verification, processor acceptance/rejection,
    forwarding attack prevention, cross-debate reuse prevention, payload edit prevention, rate
    limiting, and the new frontend flow.
