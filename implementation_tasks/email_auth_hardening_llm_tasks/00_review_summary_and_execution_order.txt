Important review corrections made to the original plan:
1. The posts table already has a user_id column (database.py line 361). No
   migration is needed. DebateEngineV2.submit_post already accepts user_id.
2. Flask-Limiter (4.1.1) is already in requirements.txt. Use it for rate
   limiting; do not invent an in-memory dict.
3. PyJWT is available via the existing flask-jwt-extended dependency. The shared
   auth helper should use `import jwt` (not flask_jwt_extended imports) because
   the email processor runs outside Flask.
4. PROCESSOR_DEST_EMAIL is currently only read by the standalone email
   processor. The Flask backend must also load it to build mailto links.
5. The auth probe for "static deployments" in the original plan referenced
   non-existent endpoints (/api/me, /api/health). Task 01 replaces this with a
   concrete fallback using existing BDA.api calls.
6. The original plan's risk note contradicted Phase 3 on Auth-Token line
   wrapping. All tasks now consistently require safe continuation/unwrapping.
7. data_bridge.js has no named `autoRedirect()` function; it is an anonymous
   IIFE at the bottom of the file. Task 01 references it by location.
8. The existing /api/email-submission-template endpoint should return 410 Gone,
   not a soft deprecation.
9. Proposals are API-only (no email mode). Task 08 confirms this and skips the
   phase rather than leaving it as open research.
10. The existing test file is tests/unit/test_email_processor.py. Extend it;
    do not create a duplicate.
