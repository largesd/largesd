TASK 09: Add Reproducible Accessibility Scan

Repo root:
/Users/jonathanleung/Documents/C++/debate_system

Goal:
Add an automated accessibility scan that can run locally and in CI, focused on
critical violations for key frontend pages.

Important correction to the original plan:
Do not use global npm installs such as npm install -g @axe-core/cli. Make the
scan reproducible from project files.

Preferred approach:
Use the existing Python Playwright dependency and add a project-local axe-core
dependency for injection.

Write scope:
- package.json and package-lock.json if no JS package files exist yet
- acceptance/run_a11y_scan.py
- .github/workflows/ci.yml
- docs/guides/TESTING.md or README.md

Implementation steps:
1. Check whether a package.json already exists. If not, create a minimal one
   for development tooling only.
2. Add axe-core as a dev dependency:
   - npm install --save-dev axe-core
3. Create acceptance/run_a11y_scan.py that:
   - starts from an already-running base URL passed with --base-url
   - uses Playwright chromium
   - injects node_modules/axe-core/axe.min.js into each page
   - runs axe.run with critical and serious impacts, or starts with critical
     only if existing pages have known violations
   - prints a concise text summary
   - exits nonzero on violations at the chosen threshold
4. Scan representative pages:
   - /
   - /login.html
   - /register.html
   - /new_debate.html
   - /admin.html
   - /appeals.html
   - /governance.html
   - /topics.html
   Pages that require auth may need either API-created auth state or a mode
   that checks the unauthenticated page shell first.
5. Add a dev_workflow command only if that matches scripts/dev_workflow.py's
   existing command style. Otherwise document the direct command.
6. Update CI:
   - run npm ci before the a11y scan
   - install Playwright chromium if the job does not already
   - start the app or reuse the acceptance job's running app pattern
   - run the a11y script
7. Document how to run it locally.

Acceptance criteria:
- The accessibility scan runs from repository dependencies, not global tools.
- CI runs the scan.
- Documentation explains the command and threshold.
- Any critical violations found during implementation are fixed or documented
  with a reason and a follow-up.
- Existing acceptance tests still pass.

Suggested commands:
- npm ci
- python -m playwright install chromium
- python scripts/dev_workflow.py acceptance --port 5080 --timeout 60
- python acceptance/run_a11y_scan.py --base-url http://127.0.0.1:5080

Guardrails:
- Do not turn this into a broad redesign task.
- Do not fail CI on moderate/minor violations initially unless the current UI
  is already clean at that threshold.
- Do not add external CDN script loading. CSP should remain self-hosted.
