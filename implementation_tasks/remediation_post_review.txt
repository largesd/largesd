CRITICAL FIX 1: CSP Blocks Inline Scripts (Frontend Broken)
================================================================================

PROBLEM
-------
Flask-Talisman CSP sets script-src: "'self'" (no unsafe-inline, no nonces).
The frontend is served via send_from_directory() in api_bp.py, so Talisman
applies CSP to all HTML responses. There are 18 inline <script> blocks across
frontend/*.html. Browsers block them -> JavaScript dead -> UI non-functional.

AFFECTED FILES
--------------
  frontend/admin.html      (2 inline <script> blocks)
  frontend/appeals.html    (2 inline <script> blocks)
  frontend/audits.html     (1 inline <script> block)
  frontend/dossier.html    (1 inline <script> block)
  frontend/evidence.html   (1 inline <script> block)
  frontend/frame-dossier.html (1 inline <script> block)
  frontend/governance.html (1 inline <script> block)
  frontend/login.html      (1 inline <script> block)
  Plus any others found via grep.

FIX: Move Inline Scripts to External JS
---------------------------------------
Step 1 - Audit all inline scripts
  grep -rn '<script>' frontend/*.html | grep -v '<script src'
  Count them. For each file, note the script content.

Step 2 - Create per-page JS files
  For each HTML file with inline scripts, create a matching .js file:
    frontend/admin.html      -> frontend/static/js/admin.js
    frontend/appeals.html    -> frontend/static/js/appeals.js
    frontend/audits.html     -> frontend/static/js/audits.js
    frontend/dossier.html    -> frontend/static/js/dossier.js
    frontend/evidence.html   -> frontend/static/js/evidence.js
    frontend/frame-dossier.html -> frontend/static/js/frame-dossier.js
    frontend/governance.html -> frontend/static/js/governance.js
    frontend/login.html      -> frontend/static/js/login.js
    ... etc for any others found in Step 1

  Rules for extraction:
    - Copy the entire inline <script>...</script> contents into the .js file.
    - Remove the IIFE wrapper if it was only there for scoping.
    - Ensure DOMContentLoaded listeners are preserved.
    - If inline scripts reference server-rendered variables (e.g.,
      window.__INITIAL_STATE__), keep those as <script> tags that ONLY set
      window variables, then load the external JS. Such "data bootstrap"
      scripts are okay if they contain no logic.

Step 3 - Update HTML to load external scripts
  Replace each inline block with:
    <script src="/static/js/PAGENAME.js" defer></script>

  Place the <script src> just before </body> to maintain load order.

Step 4 - Update api_bp.py to serve static/js/
  The current api_bp.py send_from_directory only serves frontend/*.html and
  whatever is in frontend/ via relative paths. Verify that /static/js/ URLs
  resolve correctly. If not, add a route:

    @api_bp.route("/static/js/<path:filename>")
    def serve_js(filename):
        js_dir = os.path.join(os.path.dirname(__file__), "..", "..",
                              "frontend", "static", "js")
        return send_from_directory(js_dir, filename)

  OR better: configure Flask static_folder to point at frontend/static/
  and use url_for('static', filename='js/admin.js'). However, since the app
  currently sets static_folder=None, adding a dedicated blueprint route is
  the minimal change.

Step 5 - Verify CSP compliance
  Open the app in a browser (or use curl + inspect headers).
  Confirm:
    - No CSP violations in browser DevTools console.
    - All pages load and their JS executes.
    - CSRF token cookie is still set correctly.

Step 6 - Regression tests
  Run the UI acceptance suite:
    python scripts/dev_workflow.py acceptance --port 5080 --timeout 60
  Or at minimum, manually verify:
    - Login form submits
    - Admin page loads data
    - Appeals page loads table

ACCEPTANCE CRITERIA
-------------------
  [ ] Zero inline <script> blocks containing logic in frontend/*.html
  [ ] Browser DevTools shows zero CSP violations on all pages
  [ ] All interactive features (forms, tables, modals) work
  [ ] CSRF cookie still set on HTML responses

RISK MITIGATION
---------------
  - Do NOT add 'unsafe-inline' to CSP as a shortcut. That re-opens XSS.
  - Do NOT add nonces unless you have server-side HTML templating. The
    project uses static HTML files; nonces would require Jinja2 rendering.
  - Test one page at a time; verify, then move to the next.

CRITICAL FIX 3: Mobile Tables Not Actually Applied
================================================================================

PROBLEM
-------
Task 5.4 created .table-responsive CSS rules in frontend/css/media.css,
but ZERO HTML tables use the class. Tables use .table-wrap and .table,
so they do not get the card-stacking mobile behavior.

AFFECTED FILES
--------------
  frontend/admin.html    (8 tables with class="table")
  frontend/audits.html   (7 tables with class="table")
  frontend/dossier.html  (7 tables with class="table")
  frontend/governance.html (4 tables with class="table")
  frontend/verdict.html  (6 tables with class="table")
  frontend/snapshot.html (2 tables with class="table")
  frontend/appeals.html  (1 table with class="table")
  frontend/topic.html    (1 table with class="table")

FIX
---
Option A (Minimal): Merge .table-responsive behavior into .table
  In frontend/css/media.css, change the media query block from:
    @media (max-width: 767px) {
      .table-responsive, ...
    }
  To target .table inside .table-wrap:
    @media (max-width: 767px) {
      .table-wrap .table,
      .table-wrap .table thead, ...
    }
  This applies mobile styling to ALL tables without touching HTML.

Option B (Explicit): Add table-responsive class to tables
  For each <table class="table"> that should be mobile-friendly,
  change to <table class="table table-responsive">.
  This is more explicit but touches 36 table instances.

RECOMMENDED: Option A
  It is a 3-line CSS change with zero HTML churn.

Step 1 - Edit frontend/css/media.css
  Replace all instances of:
    .table-responsive
  with:
    .table-wrap .table
  within the @media (max-width: 767px) block.

Step 2 - Verify on mobile viewport
  Open Chrome DevTools, toggle device toolbar to iPhone SE (375px width).
  Visit admin.html, audits.html, dossier.html.
  Confirm tables collapse into card-like stacked rows.

ACCEPTANCE CRITERIA
-------------------
  [ ] All tables with class="table" stack vertically on screens <= 767px
  [ ] No horizontal scrollbars on mobile viewports
  [ ] Desktop layout (> 767px) unchanged

  text = re.sub(r"<[^>]+>", "", html)
This is a fallback for plain-text extraction from email HTML bodies. While
it is NOT used for XSS sanitization, a security auditor might flag it.

FIX
---

  Replace the fallback block with:

    except Exception:
        # SECURITY NOTE: This fallback is for plain-text extraction only.
        # It strips HTML tags to produce readable text from malformed email.
        # It is NOT a sanitizer. User-facing HTML is sanitized via
        # backend.sanitize.sanitize_html (nh3-based allowlist).
        text = re.sub(r"<[^>]+", "", html)
