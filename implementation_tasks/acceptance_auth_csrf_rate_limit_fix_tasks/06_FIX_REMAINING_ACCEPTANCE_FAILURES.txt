Goal
----
Fix the remaining browser acceptance failures by bringing
acceptance/run_ui_acceptance.py back in sync with the current UI:

  - AC-1 has stale blank-state text and stale page state after creating a
    debate through the API.
  - AC-2, AC-3, AC-4, AC-6, AC-8, and AC-11 create/activate a debate through
    API helpers but then try to submit posts while still on index.html.
  - submit_post() still assumes email/GitHub mode, but the acceptance
    environment now runs in authenticated live API mode.
  - generate_snapshot() still assumes an old "Generate New Snapshot" button on
    the posting page. The current UI generates snapshots from index.html via
    #generate-snapshot-btn.


Root Cause Analysis
-------------------

AC-1:
  authenticate_browser_user() loads /index.html, then AC-1 calls
  open_new_debate_page(). With no active debate, frontend/static/js/new_debate.js
  currently renders:

    No active debate. Propose one now ->

  The harness still expects:

    Create or select a debate

  After create_debate(), the browser remains on the already-loaded
  new_debate.html document. The JS computed hasActiveDebate=false before the
  debate existed, so the post section remains disabled until the page is
  reloaded or reopened.

AC-2, AC-3, AC-4, AC-6, AC-8, AC-11:
  authenticate_browser_user() ends on /index.html.
  create_debate() performs API requests only; it does not navigate.
  submit_post() then looks for selectors that only exist on /new_debate.html:

    input[name="side"]
    #argument-topic
    #facts-input
    #inference-input

  Because Playwright is still on index.html, the first side locator times out.

submit_post() mode mismatch:
  In the current acceptance environment, the harness injects an authenticated
  user token. new_debate.js therefore enters live API mode:

    canUseLiveApiMode() -> true
    currentDebateSource -> "api"

  In API mode:

    - The submit button text is "Submit Argument".
    - The stable button ID is still #submit-post-button.
    - The success status is "Argument submitted.".
    - Posts are sent directly to /api/debate/posts.
    - Local pending email posts are not created.
    - #pending-count stays 0.
    - #pending-posts is hidden.

  The harness currently hard-codes email mode:

    page.get_by_role("button", name="Generate Email").first.click()
    expect(page.locator("#status-message")).to_contain_text("Email generated")

generate_snapshot() mode/page mismatch:
  The helper currently clicks a button named "Generate New Snapshot" and then
  asserts #status-message and #pending-count. That old flow is not present on
  frontend/new_debate.html.

  Current snapshot generation UI is on frontend/index.html:

    #generate-snapshot-btn
      text: "Generate snapshot"
      visible in live API mode when Auth.isLoggedIn() and BDA.getActiveDebateId()
      is set

    #api-status
      text transitions through "Queueing snapshot...", "Generating snapshot...",
      and "Snapshot ready"

    #badge-snapshot-id and #verdict-display
      populated after snapshot generation succeeds

  Visiting new_debate.html after create_debate() is still important because
  loadDebate() calls BDA.setActiveDebateId(currentDebateId), which gives
  index.html the local active-debate context it needs to show
  #generate-snapshot-btn.


Implementation Plan

Step 1 - Add the missing import

  At the top of acceptance/run_ui_acceptance.py, add:

    import re

  Place it with the other standard-library imports, e.g. after import json.


Step 2 - Make debate creation diagnostics stricter

  In create_debate_via_api(), do not ignore the activation response. Change:

    page.request.post(
        f"{base_url}/api/debate/{data['debate_id']}/activate",
        headers={"Authorization": f"Bearer {token}"},
    )

  to:

    activate_response = page.request.post(
        f"{base_url}/api/debate/{data['debate_id']}/activate",
        headers={"Authorization": f"Bearer {token}"},
    )
    if not activate_response.ok:
        raise AssertionError(
            "Failed to activate debate via API: "
            f"HTTP {activate_response.status}: {response_debug(activate_response)}"
        )

  This is not the primary failure, but it prevents future false diagnosis if
  debate activation breaks.


Step 3 - Add a helper for the repeated authenticated debate setup

  Add this helper near create_debate()/open_new_debate_page():

    def create_active_debate_and_open_posting(page: Page, base_url: str) -> dict[str, str]:
        debate = create_debate(page, base_url)
        open_new_debate_page(page, base_url)
        expect(page.locator("#display-resolution")).to_contain_text(debate["motion"])
        expect(page.locator("#post-access-hint")).to_contain_text("Posting is available")
        return debate

  Important details:

    - This helper assumes authenticate_browser_user(page, base_url) has already
      run for the criterion.
    - open_new_debate_page() is not just visual navigation. It lets
      new_debate.js load the active debate from the API and persist
      bda_active_debate_id through BDA.setActiveDebateId().
    - That local active-debate ID is required later by index.html snapshot
      generation.


Step 4 - Update submit_post() for both email mode and live API mode

  Change submit_post() from a role/name selector to the stable button ID:

    OLD:
      page.get_by_role("button", name="Generate Email").first.click()

    NEW:
      page.locator("#submit-post-button").click()

  Change the status assertion from an email-only assertion to a mode-aware
  assertion:

    OLD:
      expect(page.locator("#status-message")).to_contain_text("Email generated")

    NEW:
      expect(page.locator("#status-message")).to_contain_text(
          re.compile(r"Email generated|Argument submitted")

  Do not switch to a fragile text selector such as "Submit Argument"; the text
  intentionally differs by mode while #submit-post-button is stable.


Step 5 - Replace generate_snapshot() with the current UI flow

  Change generate_snapshot() to accept base_url:

    def generate_snapshot(page: Page, base_url: str) -> dict[str, str]:

  Implement it against index.html, then return to new_debate.html so existing
  post-submission call sites remain usable, especially AC-8 where another
  submit_post() happens after the first snapshot.

  Use this shape:

    def generate_snapshot(page: Page, base_url: str) -> dict[str, str]:
        page.goto(f"{base_url}/index.html", wait_until="networkidle")

        generate_button = page.locator("#generate-snapshot-btn")
        expect(generate_button).to_be_visible(timeout=SNAPSHOT_TIMEOUT_MS)
        expect(generate_button).to_be_enabled(timeout=SNAPSHOT_TIMEOUT_MS)
        generate_button.click()

        expect(page.locator("#api-status")).to_contain_text(
            "Snapshot ready", timeout=SNAPSHOT_TIMEOUT_MS
        )

        snapshot_id = (page.locator("#badge-snapshot-id").text_content() or "").strip()
        verdict = (page.locator("#verdict-display").text_content() or "").strip()

        if not snapshot_id or snapshot_id == "-" or not snapshot_id.startswith("snap_"):
            raise AssertionError("Snapshot id did not populate after generation.")
        if verdict not in {"FOR", "AGAINST", "NO VERDICT"}:
            raise AssertionError(f"Unexpected verdict value: {verdict!r}")

        open_new_debate_page(page, base_url)
        expect(page.locator("#header-snapshot")).to_contain_text(
            snapshot_id, timeout=SNAPSHOT_TIMEOUT_MS
        )

        return {"snapshot_id": snapshot_id, "verdict": verdict}

  Rationale:

    - index.html owns the current snapshot-generation button.
    - #status-message does not exist on index.html, so do not assert it there.
    - #pending-count is email-posting state, not snapshot-generation state.
    - Returning to new_debate.html keeps AC-8 compatible with another
      submit_post() after snapshot_one.

Risk Mitigation
---------------

  - The submit_post() selector change is backward-compatible because
    #submit-post-button exists in both email mode and live API mode.
  - The status regex preserves email-mode coverage while accepting the current
    authenticated API success text.
  - Conditional AC-2 pending assertions preserve the email-mode path without
    forcing live API mode to create local pending email records.
  - The setup helper localizes the required create/open/assert sequence and
    reduces the chance that future criteria create debates without navigating
    to the posting UI.
  - Snapshot generation remains browser-driven through the current UI button;
    it is not converted into a direct API-only shortcut.
  - No backend security behavior, seeded admin setup, or workflow environment
    settings need to change.
