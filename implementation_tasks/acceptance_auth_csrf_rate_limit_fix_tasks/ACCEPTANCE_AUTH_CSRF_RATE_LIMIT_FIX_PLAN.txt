Acceptance Auth / CSRF / Rate-Limit Fix Plan
============================================

Context
-------
Command:

  python scripts/dev_workflow.py acceptance --port 5080 --timeout 60

Observed failures:

  - AC-1 through AC-6 fail while registering the acceptance setup user:
      Failed to register acceptance user: HTTP 403

  - AC-7 currently passes (the browser form supplies CSRF automatically), but
    previously failed with an empty #success-alert when the UI registration
    request errored out.

  - AC-8 through AC-11 fail while registering the acceptance setup user:
      Failed to register acceptance user: HTTP 403

Note: Earlier runs also produced HTTP 429 (rate-limited) cascades for AC-8
through AC-11. The current run shows only HTTP 403 because
ENABLE_RATE_LIMITER=false is already set in scripts/dev_workflow.py, which
prevents the rate limiter from activating. The remaining root cause is purely
the missing CSRF double-submit on direct API calls.

These failures are unrelated to table-cell data-label attributes. They are in
the auth/acceptance harness and local acceptance-server setup.


Root Cause
----------
The acceptance runner has two registration paths:

  1. Direct API setup registration in acceptance/run_ui_acceptance.py.
  2. Real UI registration in AC-7 through frontend/register.html.

The direct API setup path posts to /api/auth/register without first obtaining a
CSRF cookie and without sending X-CSRF-Token. The backend CSRF middleware
correctly rejects mutating unauthenticated API requests when the double-submit
cookie/header pair is missing or mismatched.

AC-7 currently succeeds because the browser form naturally includes CSRF, but it
is fragile: if the UI registration request ever fails, the test waits 45 seconds
on an empty #success-alert instead of failing fast with the real API error.

Implementation Plan
-------------------

1. Add a CSRF-aware Playwright helper

   File:
     acceptance/run_ui_acceptance.py

   Add a helper that primes the browser context by visiting an HTML page, then
   reads the csrf_token cookie from the Playwright browser context.

   Suggested shape:

     def get_csrf_token(page: Page, base_url: str) -> str:
         page.goto(f"{base_url}/login.html", wait_until="networkidle")
         cookies = page.context.cookies(base_url)
         token = next((c["value"] for c in cookies if c["name"] == "csrf_token"), "")
         if not token:
             raise AssertionError("Could not obtain csrf_token cookie from login.html.")
         return token

   Important: Before relying on this helper, verify in
   backend/routes/api_bp.py (or wherever static HTML is served) that HTML
   responses unconditionally set the csrf_token cookie. If login.html is served
   as a raw static file without passing through the CSRF middleware, the helper
   will fail. In that case, use any endpoint or page route known to set the
   cookie (e.g. a safe GET that returns HTML).

   Use this helper before direct API calls to:

     - /api/auth/register
     - /api/auth/login

   Include the token in headers:

     headers={
         "Content-Type": "application/json",
         "X-CSRF-Token": csrf,
     }

   Critical implementation detail: the backend validates both sides of the
   double-submit pair:

     - The readable csrf_token cookie
     - The matching X-CSRF-Token request header

   Therefore the direct API call must be made from the same Playwright browser
   context that obtained the cookie. The simplest safe path is:

     - Call get_csrf_token(page, base_url).
     - Use page.request.post(...) from that same page/context.
     - Do not create a fresh playwright.request.new_context() unless you also
       manually copy the csrf_token cookie into that request context.

   Sending only the X-CSRF-Token header without the matching csrf_token cookie
   will still fail with CSRF_INVALID.

   This preserves production CSRF protection while making the acceptance harness
   behave like the real browser.

3. Seed and reuse an admin-capable acceptance user

   File:
     acceptance/run_ui_acceptance.py

   Add an explicit acceptance-admin setup path. This is required because the
   acceptance workflow uses one temporary database for the full AC-1 through
   AC-11 run, and backend/database_v3.py only auto-promotes the first
   registered user to admin:

     - The backend admin_required decorator does not treat
       ADMIN_ACCESS_MODE=authenticated as "any logged-in user can administer".
       It requires a valid JWT with is_admin=True.
     - frontend/static/js/admin.js also has a pre-paint route gate that
       redirects away from admin.html unless localStorage user.is_admin is true.
     - Therefore registering a fresh non-admin user for every criterion will
       make later admin setup/API/UI flows fail after the CSRF problem is fixed.

   Required implementation:

     - Create a stable per-run acceptance admin identity, for example:

         ACCEPTANCE_ADMIN_EMAIL = f"acceptance_admin_{run_suffix}@example.com"
         ACCEPTANCE_ADMIN_PASSWORD = "AcceptancePass123"
         ACCEPTANCE_ADMIN_DISPLAY_NAME = f"Acceptance Admin {run_suffix[-6:]}"

       Store the run_suffix once at process startup, not once per criterion.

     - Seed this admin user before the criterion loop starts in main(), using a
       short-lived Playwright context/page and the CSRF-aware register/login
       helper from step 1.

       This matters if the criterion order changes. If AC-7's UI registration
       were ever run first, that UI user could become the first database user,
       and the later seeded setup account would be non-admin.

     - Cache the full auth payload for the seeded user in process memory:

         {
             "access_token": ...,
             "user_id": ...,
             "email": ...,
             "display_name": ...,
             "is_admin": True,
         }

     - Assert immediately that the seeded auth payload has is_admin=True. If it
       is false, fail with a detailed error that includes the register/login
       response body. Do not silently continue as a non-admin user, because that
       only defers the failure to AC-5 or AC-9.

     - Change authenticate_browser_user(page, base_url) to reuse this cached
       admin payload instead of registering a new user per criterion. It should
       inject both the token and the complete user object into the current
       browser context:

         page.goto(f"{base_url}/index.html", wait_until="networkidle")
         page.evaluate(
             """(auth) => {
                 localStorage.setItem("access_token", auth.access_token);
                 localStorage.setItem("user", JSON.stringify({
                     user_id: auth.user_id,
                     email: auth.email,
                     display_name: auth.display_name,
                     is_admin: auth.is_admin,
                 }));
             }""",
             admin_auth,
         )

     - Keep AC-7's register_user_via_ui() and login_user_via_ui() as real
       self-service UI flows with unique non-admin users. AC-7 does not require
       admin privileges and should continue to exercise the public auth UI.

   Expected improvement:

     - AC-1 through AC-6 and AC-8 through AC-11 get a deterministic admin-capable
       setup identity.
     - AC-5 can pass frontend/static/js/admin.js's localStorage user.is_admin
       gate before loading admin.html.
     - AC-9 can call /api/admin/debate-proposals/<proposal_id>/accept with a
       token whose JWT includes is_admin=True.


4. Make acceptance startup reject stale servers

   File:
     scripts/dev_workflow.py

   Before starting the temporary server in run_acceptance(), check whether the
   selected host/port can be bound by the new server.

   If the port is occupied, fail fast with a clear message:

     Port 5080 is already in use. Stop the existing server or choose a different
     port with --port.

   Why this matters:

     A stale server on port 5080 can make the workflow believe the new temporary
     server is healthy, while the acceptance suite is actually talking to an old
     process with live rate-limit state.

   Implementation options:

     - Add a small bind-based socket helper such as
       assert_port_available(host, port).
     - Implement it by attempting socket.bind((host, port)) and raising a clear
       RuntimeError on EADDRINUSE / OSError.
     - Use socket.AF_INET for the current DEFAULT_HOST value of 127.0.0.1.
     - Use a context manager so the socket is always closed immediately.
     - Set SO_REUSEADDR before bind to avoid false positives from recently
       closed listeners on platforms where that matters.
     - Prefer the bind check over a connect-only check. A connect check only
       tells you whether something is accepting connections; the actual
       workflow requirement is whether the new Flask process can bind the port.
     - Call it in run_acceptance() before deleting the temporary database and
       before subprocess.Popen().
     - Optionally reuse it for run_smoke().

   Suggested shape:

     def assert_port_available(host: str, port: int) -> None:
         with socket.socket(socket.AF_INET, socket.SOCK_STREAM) as sock:
             sock.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
             try:
                 sock.bind((host, port))
             except OSError as exc:
                 raise RuntimeError(
                     f"Port {port} is already in use on {host}. Stop the existing "
                     "server or choose a different port with --port. If a previous "
                     "run just exited, wait a moment for the OS to release the port."
                 ) from exc

   Important: Perform the socket check using the same host interface that the
   acceptance server will bind to (e.g. 127.0.0.1). On some systems a port in
   TIME_WAIT from a previous run may briefly remain occupied; the error message
   should mention this possibility.

   Important: Keep wait_for_health(base_url, args.timeout, process) checking
   process.poll(). The bind check catches stale servers before startup, but the
   Flask process can still exit after Popen if something changes between the
   check and the actual bind.

   Important ordering detail: run the port check before deleting
   debate_system_acceptance_<port>.db. If a stale acceptance server is still
   running on the same port and using that temp DB path, deleting the DB first
   can disturb the stale process before the workflow reports the real issue.


5. Harden the acceptance server environment

   File:
     scripts/dev_workflow.py

   In run_acceptance(), keep:

     ENABLE_RATE_LIMITER=false

   Also explicitly set:

     ENV=development
     ALLOWED_ORIGINS=
     ADMIN_ACCESS_MODE=authenticated
     ADMIN_USER_EMAILS=
     ADMIN_USER_IDS=

   Important: Before setting ALLOWED_ORIGINS= (empty string), verify in
   backend/app_v3.py that the CORS layer handles an empty value gracefully
   (e.g. falls back to same-origin or localhost). If the backend splits the
   value on commas and produces a list containing an empty string, it may
   reject same-origin requests. In that case, set an explicit value matching
   the acceptance base URL, such as ALLOWED_ORIGINS=http://127.0.0.1:5080.

   Current code note: backend/app_v3.py already filters empty origin entries and
   disables CORS when ALLOWED_ORIGINS is explicitly blank. That is safe for this
   same-origin acceptance workflow because browser requests originate from the
   same host and port as the API.

   Important: ADMIN_ACCESS_MODE must be pinned for acceptance. The workflow
   currently inherits .env through project_env(), and admin-only acceptance
   flows call endpoints such as /api/debates and
   /api/admin/debate-proposals/<proposal_id>/accept. If a developer's local env
   leaves ADMIN_ACCESS_MODE unset or sets it to restricted, the CSRF fix will
   simply expose the next 403 failure. Pinning authenticated mode is necessary
   but not sufficient: the seeded admin user from step 3 must still be the first
   registered user in the isolated acceptance database and must be reused for
   setup/admin flows so its JWT and localStorage user payload carry
   is_admin=True.

   Do not set DISABLE_JOB_WORKER=1 for acceptance runs. The acceptance criteria
   exercise snapshot generation through the browser, and /api/debate/snapshot
   enqueues background jobs that must be processed by the JobWorker before the UI
   can report completion. If a developer's local .env or shell sets
   DISABLE_JOB_WORKER=1, explicitly override it to an empty value or remove it
   from the subprocess environment before starting the acceptance server.

   Rationale:

     - ENV=development ensures local HTTP CSRF cookies are not marked Secure.
     - ALLOWED_ORIGINS= keeps same-origin local acceptance predictable.
     - ADMIN_ACCESS_MODE=authenticated prevents local .env restricted allowlists
       from breaking the seeded acceptance admin account.
     - ENABLE_RATE_LIMITER=false prevents test setup traffic from exhausting
       global mutating API limits.
     - Keeping the job worker enabled prevents AC-3, AC-6, AC-8, and AC-11 from
       timing out while polling queued snapshot jobs.


6. Make AC-7 fail on the real registration or login error

   File:
     acceptance/run_ui_acceptance.py

   In register_user_via_ui(), wrap the submit click with a response wait for
   /api/auth/register.

   Suggested flow:

     with page.expect_response(
         lambda r: "/api/auth/register" in r.url and r.request.method == "POST"
     ) as response_info:
         page.locator("#submit-btn").click()
     response = response_info.value

   Then:

     - If response.status is not 200 or 201, read the response body and
       #error-alert text and raise an AssertionError immediately.
     - Only assert #success-alert contains "Account created" after confirming
       the network response succeeded.

   Apply the same pattern to login_user_via_ui() for /api/auth/login:

     with page.expect_response(
         lambda r: "/api/auth/login" in r.url and r.request.method == "POST"
     ) as response_info:
         page.locator("#submit-btn").click()
     response = response_info.value

   Then:

     - If response.status is not 200, read the response body and #error-alert
       text and raise an AssertionError immediately.
     - Only assert #success-alert contains "Login successful" after confirming
       the network response succeeded.

   Important: Ensure the form is fully loaded and the submit button is enabled
   before clicking, to avoid races where expect_response misses an in-flight
   request.

   Expected improvement:

     AC-7 should no longer wait 45 seconds on an empty success alert when the
     actual registration or login problem is a 403, 429, or other API response.


7. Keep backend CSRF behavior unchanged

   Files:
     backend/utils/middleware.py
     backend/routes/auth_bp.py

   Do not exempt /api/auth/register or /api/auth/login from CSRF protection.
   The backend behavior is correct:

     - Safe methods skip CSRF.
     - Bearer-authenticated mutating API calls skip CSRF.
     - Unauthenticated mutating API calls require matching csrf_token cookie and
       X-CSRF-Token header.

   The fix belongs in the acceptance client and workflow isolation, not in a
   production security bypass.
