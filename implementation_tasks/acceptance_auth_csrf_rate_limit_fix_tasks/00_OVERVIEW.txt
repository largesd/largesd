Acceptance Auth / CSRF / Rate-Limit Fix Task Overview
=====================================================

Canonical source:
  ACCEPTANCE_AUTH_CSRF_RATE_LIMIT_FIX_PLAN.txt

This folder splits the master plan into implementation-sized task files for an
LLM implementer. Keep the master file as the source of truth, but use these
numbered files to execute the work in a safer order.


Current Failure Pattern
-----------------------
Command:

  python scripts/dev_workflow.py acceptance --port 5080 --timeout 60

Observed current result:

  - AC-1 through AC-6 fail while registering the setup user:
      Failed to register acceptance user: HTTP 403

  - AC-7 currently passes because the real browser form gets and sends CSRF
    correctly, but it is fragile and can wait on an empty #success-alert when
    the registration or login API call actually failed.

  - AC-8 through AC-11 fail while registering the setup user:
      Failed to register acceptance user: HTTP 403

Earlier runs also produced HTTP 429 cascades for AC-8 through AC-11. The
current code already sets ENABLE_RATE_LIMITER=false for acceptance, so the
remaining main setup failure is missing CSRF double-submit handling in the
direct Playwright API setup path.


Root Causes
-----------
1. Direct API setup registration in acceptance/run_ui_acceptance.py posts to
   /api/auth/register without first obtaining a csrf_token cookie and without
   sending the matching X-CSRF-Token header.

2. The direct API setup path registers a fresh user per criterion. After CSRF
   is fixed, that creates a second failure mode: only the first user in the
   shared acceptance database is auto-admin, while admin_required in
   ADMIN_ACCESS_MODE=authenticated still requires JWT is_admin=True.

3. frontend/static/js/admin.js also gates admin.html before paint and redirects
   unless localStorage.user.is_admin is true.

4. A stale server on the selected acceptance port can make the workflow talk to
   an old process with old env/rate-limit/database state.

5. AC-7's UI helper waits for success text instead of first inspecting the real
   /api/auth/register or /api/auth/login response.


Non-Negotiable Constraints
--------------------------
Do not weaken backend security:

  - Do not exempt /api/auth/register from CSRF.
  - Do not exempt /api/auth/login from CSRF.
  - Do not change backend/utils/middleware.py to bypass CSRF for unauthenticated
    mutating API calls.
  - Bearer-authenticated mutating API calls should continue skipping CSRF.

Direct setup auth must behave like a browser:

  - Visit an HTML page first to receive the readable csrf_token cookie.
  - Read the csrf_token cookie from the same Playwright browser context.
  - Send X-CSRF-Token with the exact cookie value.
  - Use page.request.post(...) from the same page/context that has the cookie.
  - Do not create a fresh playwright.request.new_context() unless the cookie is
    explicitly copied into that request context.

Acceptance env must be deterministic:

  - Keep ENABLE_RATE_LIMITER=false.
  - Set ENV=development so local HTTP CSRF cookies are not Secure-only.
  - Set ALLOWED_ORIGINS= for same-origin acceptance.
  - Set ADMIN_ACCESS_MODE=authenticated.
  - Clear ADMIN_USER_EMAILS and ADMIN_USER_IDS.
  - Do not run acceptance with DISABLE_JOB_WORKER=1. If a local .env could set
    it to 1, explicitly override it to an empty value or remove it from the
    subprocess env before starting the server. Snapshot criteria need the
    JobWorker.

Admin setup must be deterministic:

  - Seed a stable per-run admin user before the criterion loop starts.
  - Ensure that seeded user is the first registered user in the temporary DB.
  - Assert the auth payload includes is_admin=True.
  - Reuse that admin token/user payload for setup/admin criteria.
  - Inject localStorage "user" with is_admin: true, not just access_token.
  - Keep AC-7 as a real non-admin self-service UI registration/login flow.


Recommended Task Order
----------------------
1. 01_CSRF_AUTH_HELPER.txt
   Add CSRF-aware direct API registration/login helpers and better API failure
   messages.

2. 02_SEEDED_ADMIN_USER.txt
   Seed one admin-capable acceptance user before AC execution and reuse its
   token/user payload for setup/admin flows.

3. 03_DEV_WORKFLOW_ENV_PORT_GUARD.txt
   Add deterministic acceptance server env and a bind-based stale-server guard.

4. 04_AC7_UI_ERROR_REPORTING.txt
   Make AC-7 fail quickly with real registration/login response bodies.

5. 05_TESTS_AND_VERIFICATION.txt
   Add focused regression coverage and run the verification commands.

Implementation note:
  Tasks 01 and 02 both edit acceptance/run_ui_acceptance.py and should be done
  by the same implementer or very carefully sequenced. Do not assign them to two
  agents with overlapping write scope.
