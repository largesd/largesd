Task 02: Seed and Reuse an Admin-Capable Acceptance User
========================================================

Primary file:
  acceptance/run_ui_acceptance.py

Related files for behavior verification:
  backend/database_v3.py
  backend/utils/decorators.py
  frontend/static/js/admin.js


Goal
----
Make acceptance setup/admin flows deterministic by seeding exactly one per-run
admin user before AC-1 through AC-11 execute, then reusing that user's token and
complete user payload for setup/admin criteria.


Why This Is Required
--------------------
The acceptance workflow uses one temporary database for the whole run.
backend/database_v3.py auto-promotes only the first registered user:

  "First registered user becomes admin."

backend/utils/decorators.py does not treat ADMIN_ACCESS_MODE=authenticated as
"any logged-in user can administer". It still requires:

  g.user.get("is_admin") == True

frontend/static/js/admin.js also redirects away from admin.html before paint
unless localStorage user.is_admin is true.

Therefore, after Task 01 fixes CSRF, registering a fresh user per criterion can
make only AC-1's user admin. Later criteria can fail with 403s or admin.html
redirects.


Required Implementation
-----------------------
Add a stable per-run admin identity. Generate the suffix once at process startup,
not once per criterion:

  ACCEPTANCE_RUN_SUFFIX = unique_suffix()
  ACCEPTANCE_ADMIN_EMAIL = f"acceptance_admin_{ACCEPTANCE_RUN_SUFFIX}@example.com"
  ACCEPTANCE_ADMIN_PASSWORD = "AcceptancePass123"
  ACCEPTANCE_ADMIN_DISPLAY_NAME = f"Acceptance Admin {ACCEPTANCE_RUN_SUFFIX[-6:]}"

Add a module-level cache, for example:

  ACCEPTANCE_ADMIN_AUTH: dict[str, object] | None = None

Seed the admin user before the criterion loop starts in main():

  - Start Playwright.
  - Launch the browser.
  - Create a short-lived context/page.
  - Set page default timeout.
  - Register or log in the acceptance admin using the CSRF-aware helper from
    Task 01.
  - Cache the full auth payload.
  - Close the seed context.
  - Then begin the AC-1 through AC-11 criterion loop.

This ordering matters. If AC-7's UI registration ever runs first due to a future
criterion-order change, AC-7's non-admin user could become the first database
user and the setup account would not be admin.


Auth Payload Requirements
-------------------------
Cache the full payload, not just the access token:

  {
      "access_token": "...",
      "user_id": "...",
      "email": "...",
      "display_name": "...",
      "is_admin": True,
  }

Immediately assert:

  if admin_auth.get("is_admin") is not True:
      raise AssertionError(
          "Seeded acceptance admin user is not admin. "
          f"Auth payload: {admin_auth}"
      )

If seeding fails, include the register/login response body from response_debug().
Do not silently continue as a non-admin user.


Update authenticate_browser_user
--------------------------------
Change authenticate_browser_user(page, base_url) so it reuses the cached admin
payload instead of registering a new user per criterion.

It should inject both localStorage entries:

  page.goto(f"{base_url}/index.html", wait_until="networkidle")
  page.evaluate(
      """(auth) => {
          localStorage.setItem("access_token", auth.access_token);
          localStorage.setItem("user", JSON.stringify({
              user_id: auth.user_id,
              email: auth.email,
              display_name: auth.display_name,
              is_admin: auth.is_admin,
          }));
      }""",
      admin_auth,
  )

Return a user dict compatible with current callers:

  {
      "user_id": admin_auth["user_id"],
      "email": admin_auth["email"],
      "display_name": admin_auth["display_name"],
  }


AC-7 Must Stay Separate
----------------------
Do not convert AC-7 to use the seeded admin user.

Keep register_user_via_ui() and login_user_via_ui() as real browser UI flows
with a unique user:

  - AC-7 should exercise register.html and login.html.
  - AC-7 does not need admin privileges.
  - AC-7's user can be non-admin.


Expected Behavior
-----------------
After this task:

  - AC-1 through AC-6 and AC-8 through AC-11 use a deterministic admin-capable
    setup user.
  - AC-5 passes frontend/static/js/admin.js's localStorage user.is_admin gate.
  - AC-9 can call /api/admin/debate-proposals/<proposal_id>/accept with a JWT
    that includes is_admin=True.


Regression Signals
------------------
Add or update acceptance-harness coverage if practical:

  - The seeded admin payload includes is_admin=True.
  - authenticate_browser_user injects localStorage user.is_admin=true.

The full acceptance workflow is the main end-to-end validation:

  python scripts/dev_workflow.py acceptance --port 5080 --timeout 60
