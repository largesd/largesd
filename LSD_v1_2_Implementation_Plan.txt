WORKFLOW INTEGRATION (MANDATORY)
================================================================================

This roadmap must be executed through the checkpointed workflow in
docs/workflow/WORKFLOW.md. The roadmap below is the backlog; it is not
permission to execute multiple phases without human checkpoints.

Workflow Rules:
- Only one workflow phase may be active at a time.
- Treat each step below as input to the current workflow phase, not as an
  instruction to execute the whole plan in one pass.
- Before branching into new work, log the human instruction:
    ./wf log-human-prompt "..."
- If the active workflow goal no longer matches the next slice, log the change:
    ./wf change-requirement "Prioritize real OpenRouter integration and defer final model selection until smoke-test criteria are defined"
- Consult workflow_state/scaffolding/* before broad repository reads.
- Produce the required artifact(s) for the current phase, then stop at the
  checkpoint and wait for human approval.
- Advance only with:
    ./wf resume --human-note "..."
- If revisions are requested but scope has not changed, stay in-phase with:
    ./wf reject --human-note "..."

Recommended Mapping From This Roadmap To Workflow Phases:
1. System design:
   - choose the next bounded slice from this roadmap
   - define success criteria, risks, and acceptance target
2. Module breakdown:
   - map the slice to exact files, tests, and dependencies
3. Module design:
   - write or refresh the implementation note for the slice
4. Module implementation:
   - implement only the approved slice
5. Module testing:
   - run targeted tests and manual smoke checks for the slice
6. Integration testing:
   - run scripts/dev_workflow.py check, acceptance, and summarize residual risk

Current Planning Note:
- If workflow_state/state.json still points at the earlier frame-transparency /
  dossier goal, use ./wf change-requirement before starting the OpenRouter
  cutover. Do not silently switch the active slice.

STEP 1.2: IMPLEMENT MANDATORY NORMATIVE SYMMETRY TESTS (LSD §14)
--------------------------------------------------------------------------------
Objective: Add the mandatory role-swap and comparable-case tests for normative
premise assessment.

Files to Modify:
  - backend/scoring_engine.py
  - backend/llm_client.py
  - backend/lsd_v1_2.py (diagnostic helpers if needed)
  - frontend/dossier.html (surface results)

Implementation Plan:
  1. In scoring_engine.py, add run_normative_symmetry_tests():
     - Accept a normative premise text and active Frame.
     - ROLE-SWAP TEST: Swap the positions of the primary actors/agents in the
       premise text (e.g., "Government should tax X" -> "Citizens should tax
       Government"). Re-evaluate q = P(acceptable | Frame).
     - COMPARABLE-CASE TEST: Construct a structurally analogous premise in a
       different domain. Re-evaluate q.
     - Compute delta_q = |q_original - q_swapped| and |q_original - q_comparable|.
  2. In llm_client.py, add judge_normative_symmetry() prompt that returns:
     {"q_original": float, "q_swapped": float, "q_comparable": float,
      "swap_description": str, "comparable_case": str, "delta_q": float}
  3. Integrate into compute_debate_scores() so each normative premise includes
     a symmetry_test_results field.
  4. Publish test descriptions and deltas in the decision dossier.
  5. In frontend/dossier.html, add a "Normative Symmetry" section showing:
     - swap description
     - delta_q
     - flag if delta_q > 0.2 (indicates frame-sensitive premise)

Testing:
  - Add test_symmetry_tests_detect_frame_bias() to test_lsd_v1_2_contracts.py.
  - Verify that a premise with clear actor asymmetry produces delta_q > 0.

STEP 2.3: DETERMINISTIC REPLAY VERIFICATION
--------------------------------------------------------------------------------
Objective: Prove that a snapshot can be deterministically reproduced from its
published inputs and parameters.

Files to Modify:
  - backend/debate_engine_v2.py
  - backend/lsd_v1_2.py
  - backend/app_v3.py
  - frontend/snapshot.html

Implementation Plan:
  1. In debate_engine_v2.py, build a verify_snapshot(snapshot_id) function that:
     a. Loads the stored replay_manifest and input_bundle.
     b. Re-runs the exact pipeline with the same seeds, same template version,
        same selection formula, and same frame.
     c. Compares the new output_bundle hash to the stored output hash.
     d. Returns {verified: true/false, hash_match: true/false, delta: {...}}
  2. Expose POST /api/admin/snapshots/<snapshot_id>/verify (admin only).
  3. In frontend/snapshot.html, add a "Verify Replay" button for admins that:
     - Calls the verify endpoint
     - Shows a green "Replay Verified" badge if hash_match=true
     - Shows a red warning with delta details if mismatch
  4. Add a nightly or on-demand verification job to the job queue.

Definition of Done:
  - At least 3 consecutive snapshots verify successfully in a row.
  - The verify button is visible and functional in the admin view of snapshot.
