================================================================================
EMAIL POSTING MECHANISM — IMPLEMENTATION PLAN
Blind Debate Adjudicator / debate_system
================================================================================

OVERVIEW
--------
This plan implements six targeted improvements to the email submission and
GitHub publishing pipeline. Each item is broken into concrete file changes,
logic additions, and test steps. Estimated total effort: 1–2 days for a single
developer familiar with the codebase.

PREREQUISITES
-------------
• Python 3.11+ with venv activated
• Existing test suite runs green: pytest test_email_processor.py
• Frontend served from /frontend/ (static files, no build step required)
• GitHub token with repo scope for integration testing

SUGGESTED IMPLEMENTATION ORDER
------------------------------
1. Update .env.example (no risk, sets the stage)
2. Improve section extraction (parser improvement, affects downstream logic)
3. Add HTML-to-text fallback (parser robustness)
4. Deduplicate by Submission-ID (data integrity)
5. Unify email body generation (frontend/backend contract)
6. Make GitHub publish atomic / retriable (system reliability)

RECOMMENDATION 1: HTML-to-Text Fallback in email_processor
================================================================================

OBJECTIVE
---------
Prevent submission failures when participants send HTML-only emails. Currently
_get_text_body() returns None if text/plain is absent, causing the processor
to reject the email.

FILES TO MODIFY
---------------
• backend/email_processor.py
• requirements.txt (optional, see below)
• test_email_processor.py

IMPLEMENTATION STEPS
--------------------
1. Add a new static helper method to EmailProcessor:

   @staticmethod
   def _get_html_body(msg: email.message.Message) -> Optional[str]:
       """Extract HTML body and convert to plain text."""
       html = None
       if msg.is_multipart():
           for part in msg.walk():
               if part.get_content_type() == "text/html":
                   payload = part.get_payload(decode=True)
                   if payload:
                       charset = part.get_content_charset() or "utf-8"
                       html = payload.decode(charset, errors="replace")
                       break
       else:
           if msg.get_content_type() == "text/html":
               payload = msg.get_payload(decode=True)
               if payload:
                   charset = msg.get_content_charset() or "utf-8"
                   html = payload.decode(charset, errors="replace")
       return html

2. Add a lightweight HTML-to-text converter. Two options:

   OPTION A — No new dependencies (preferred for now):
   Add a regex-based stripper inside email_processor.py:

   import re
   from html.parser import HTMLParser

   class _HTMLToText(HTMLParser):
       def __init__(self):
           super().__init__()
           self.text = []
       def handle_data(self, data):
           self.text.append(data)
       def get_text(self):
           return re.sub(r'\n\s*\n+', '\n\n', ''.join(self.text)).strip()

   @staticmethod
   def _html_to_text(html: str) -> str:
       parser = _HTMLToText()
       try:
           parser.feed(html)
           return parser.get_text()
       except Exception:
           # ultimate fallback: regex strip
           text = re.sub(r'<[^>]+>', '', html)
           return re.sub(r'\n\s*\n+', '\n\n', text).strip()

   OPTION B — Add html2text dependency:
   In requirements.txt add: html2text>=2024.0
   Then use html2text.html2text(html).
   (Only do this if the team is comfortable adding a dependency.)

3. Modify _get_text_body() to fall back to HTML extraction:

   @staticmethod
   def _get_text_body(msg: email.message.Message) -> Optional[str]:
       # existing plain-text extraction logic ...
       text = EmailProcessor._extract_plain_text(msg)
       if text:
           return text
       html = EmailProcessor._get_html_body(msg)
       if html:
           return EmailProcessor._html_to_text(html)
       return None

   (Rename the existing logic to _extract_plain_text() or keep inline.)

4. Add test coverage in test_email_processor.py:
   • test_get_text_body_prefers_plain_text
   • test_get_text_body_falls_back_to_html
   • test_get_text_body_returns_none_for_image_only_email

ACCEPTANCE CRITERIA
-------------------
• An email with Content-Type: text/html and no text/plain part is parsed
  successfully.
• HTML tags are stripped; entities like &lt; remain as <.
• Plain text is still preferred when both parts exist.

RECOMMENDATION 2: Deduplicate by Submission-ID
================================================================================

OBJECTIVE
---------
Prevent duplicate posts if the same email is processed twice (e.g., IMAP sync
issue, processor restart, or mark_processed=false).

FILES TO MODIFY
---------------
• backend/email_processor.py
• backend/database_v3.py (or backend/database.py, whichever is canonical)
• test_email_processor.py

IMPLEMENTATION STEPS
--------------------
1. Add a query method to the Database class. In backend/database_v3.py:

   def get_post_by_submission_id(self, submission_id: str) -> Optional[Dict]:
       """Return a post row if submission_id already exists."""
       cursor = self.conn.cursor()
       cursor.execute(
           "SELECT post_id, debate_id, timestamp FROM posts WHERE submission_id = ?",
           (submission_id,)
       )
       row = cursor.fetchone()
       if row:
           return {
               "post_id": row[0],
               "debate_id": row[1],
               "timestamp": row[2],
           }
       return None

   NOTE: If the posts table does not have a submission_id column, add it:

   immediately after parsing succeeds:

        # Parse submission

            submission = self.parser.parse_body(body, submitter_email, subject)
        except EmailSubmissionError as exc:

RECOMMENDATION 3: Atomic / Retriable GitHub Publishing
================================================================================

OBJECTIVE
---------
If GitHub publishing fails after the local database has already been updated,
the server state diverges from GitHub. We need either (a) retry with backoff,
or (b) queue failed publishes for the next poll cycle.

FILES TO MODIFY
---------------
• backend/email_processor.py
• backend/github_publisher.py (minor)
• backend/database_v3.py
• test_email_processor.py

IMPLEMENTATION STEPS
--------------------
1. Add a "failed publishes" queue table in the database. In database_v3.py:

   def _ensure_failed_publish_queue(self):
       cursor = self.conn.cursor()
       cursor.execute('''
           CREATE TABLE IF NOT EXISTS failed_publishes (
               id INTEGER PRIMARY KEY AUTOINCREMENT,
               debate_id TEXT NOT NULL,
               snapshot_id TEXT,
               payload_json TEXT NOT NULL,
               commit_message TEXT NOT NULL,
               retry_count INTEGER DEFAULT 0,
               last_error TEXT,
               created_at TEXT NOT NULL,
               next_retry_at TEXT
           )
       ''')
       self.conn.commit()

2. Modify _process_message() in email_processor.py. Wrap the GitHub publish
   block in a retry loop with exponential backoff:

   MAX_RETRIES = 3
   BASE_DELAY = 5  # seconds

   for attempt in range(1, MAX_RETRIES + 1):
       try:
           bundle = self.results_builder.build_bundle(...)
           result = self.publisher.publish_json(bundle, bundle["commit_message"])
           print(f"[EmailProcessor] Published to GitHub: {result.commit_sha}")
           break  # success
       except GitHubPublishError as exc:
           print(f"[EmailProcessor] GitHub publish attempt {attempt} failed: {exc}")
           if attempt == MAX_RETRIES:
               # Queue for later retry
               self._queue_failed_publish(debate_id, snapshot, bundle, str(exc))
               self._send_ack(submitter_email, subject, False,
                   f"GitHub publish failed after {MAX_RETRIES} attempts. It will be retried automatically.")
               return
           time.sleep(BASE_DELAY * (2 ** (attempt - 1)))
       except Exception as exc:
           # Unexpected error, also queue
           self._queue_failed_publish(debate_id, snapshot, bundle, str(exc))
           self._send_ack(...)
           return

3. Add _queue_failed_publish() to EmailProcessor:

   def _queue_failed_publish(self, debate_id, snapshot, bundle, error_msg):
       if not hasattr(self.debate_engine.db, 'queue_failed_publish'):
           return
       self.debate_engine.db.queue_failed_publish(
           debate_id=debate_id,
           snapshot_id=snapshot.get('snapshot_id') if snapshot else None,
           payload_json=json.dumps(bundle),
           commit_message=bundle["commit_message"],
           last_error=error_msg,
       )

4. Add a "drain failed publishes" step at the start of each poll cycle:

   def _poll_once(self, search_criteria: str = "UNSEEN") -> None:
       # Before processing new mail, retry any queued failures
       self._retry_failed_publishes()
       # ... existing IMAP logic ...

   def _retry_failed_publishes(self):
       if not hasattr(self.debate_engine.db, 'get_failed_publishes'):
           return
       pending = self.debate_engine.db.get_failed_publishes(limit=10)
       for item in pending:
           try:
               payload = json.loads(item['payload_json'])
               result = self.publisher.publish_json(
                   payload=payload,
                   commit_message=item['commit_message'],
               )
               self.debate_engine.db.mark_publish_success(item['id'])
               print(f"[EmailProcessor] Retried publish succeeded: {result.commit_sha}")
           except Exception as exc:
               self.debate_engine.db.increment_publish_retry(
                   item['id'], str(exc)
               )
               print(f"[EmailProcessor] Retried publish failed again: {exc}")

5. Add DB helper methods:
   • queue_failed_publish()
   • get_failed_publishes(limit)
   • mark_publish_success(id)
   • increment_publish_retry(id, error)

6. Add tests:
   • test_github_publish_retries_on_failure — mock publisher to fail twice,
     then succeed; assert final success.
   • test_github_publish_queues_after_max_retries — mock publisher to always
     fail; assert entry in failed_publishes table.
   • test_retry_failed_publishes_on_next_poll — insert a failed publish,
     run _poll_once, assert it gets published.

ACCEPTANCE CRITERIA
-------------------
• A transient GitHub API failure is retried 3 times with exponential backoff.
• After max retries, the bundle is queued in SQLite and retried on the next
  poll cycle without re-processing the email.
• If the retry succeeds later, the queue entry is removed.

RECOMMENDATION 4: Unify Email Body Generation
================================================================================

OBJECTIVE
---------
Eliminate format drift between frontend (data_bridge.js inline parser) and
backend (email_submission_parser.py). The frontend should generate the body
using the same canonical template as the backend.

FILES TO MODIFY
---------------
• backend/email_submission_parser.py
• backend/app_v3.py
• frontend/static/js/data_bridge.js
• test_email_processor.py

IMPLEMENTATION STEPS
--------------------
1. Export the body-building schema from the backend as a lightweight endpoint.
   In backend/app_v3.py, add:

   @app.route('/api/email-submission-template', methods=['GET'])
   def get_email_submission_template():
       """Return the canonical email body template and expected fields."""
       return jsonify({
           "version": "BDA Submission v1",
           "required_headers": [
               "Debate-ID", "Resolution", "Submission-ID",
               "Submitted-At", "Position", "Topic-Area"
           ],
           "body_sections": ["Facts", "Inference", "Counter-Arguments"],
           "example": EmailSubmissionParser().build_email_body(
               debate_id="DEBATE_ID",
               resolution="RESOLUTION_TEXT",
               side="FOR",
               topic_id="t1",
               facts="Your facts here.",
               inference="Your inference here.",
               counter_arguments="Optional counter-arguments."
           ),
       })

2. Expose build_email_body() more cleanly in email_submission_parser.py.
   It already exists as a public method; just verify the class can be imported
   standalone without side effects.

3. Refactor frontend data_bridge.js to remove the inline parser object and
   instead build the body using a canonical template function that mirrors
   the backend exactly. Two strategies:

   STRATEGY A — Fetch template from API (preferred when API is available):
   async function fetchTemplate() {
       try {
           const res = await fetch('/api/email-submission-template');
           if (res.ok) return (await res.json()).example;
       } catch (e) { /* fall through */ }
       return null;
   }

   STRATEGY B — Inline canonical builder that is a line-for-line JS port of
   email_submission_parser.py::build_email_body():

   buildEmailBody(debate_id, resolution, side, topic_id, facts, inference, counterArguments) {
       const submissionId = self.crypto?.randomUUID
           ? self.crypto.randomUUID()
           : 'sub-' + Date.now();
       const submittedAt = new Date().toISOString();
       const lines = [
           'BDA Submission v1',
           `Debate-ID: ${debate_id}`,
           `Resolution: ${resolution}`,
           `Submission-ID: ${submissionId}`,
           `Submitted-At: ${submittedAt}`,
           `Position: ${side}`,
           `Topic-Area: ${topic_id}`,
           '',
           'Facts:',
           facts,
           '',
           'Inference:',
           inference,
       ];
       if (counterArguments) {
           lines.push('', 'Counter-Arguments:', counterArguments);
       }
       return lines.join('\n');
   }

   Choose Strategy B for offline-first GitHub mode, but keep Strategy A as a
   live sync option. The critical fix is: delete the old inline `parser` object
   in data_bridge.js and replace it with the canonical `buildEmailBody` method.

4. Add a shared schema file both sides can reference:
   Create docs/schema/email_submission_v1.md documenting the exact format,
   so future maintainers know both sides must stay in sync.

5. Add a regression test:
   • test_frontend_and_backend_body_format_match — build a body in Python and
     in JS (via a Node script or string comparison), assert they are identical
     modulo UUID and timestamp.

ACCEPTANCE CRITERIA
-------------------
• The frontend buildEmailBody() and backend build_email_body() produce identical
  line ordering and spacing.
• A backend parser unit test can parse a body generated by the frontend builder.
• There is a single documented source of truth for the format.

RECOMMENDATION 6: Improve Section Extraction
================================================================================

OBJECTIVE
---------
Make the regex that extracts Facts, Inference, and Counter-Arguments more
tolerant of single newlines, trailing whitespace, and variations in header
formatting. Optionally introduce a BDA Submission v2 format using a more
structured delimiter.

FILES TO MODIFY
---------------
• backend/email_submission_parser.py
• test_email_processor.py

IMPLEMENTATION STEPS
--------------------
1. Refactor _extract_section() to be more tolerant. Replace the current
   implementation with a two-pass strategy:

   def _extract_section(self, text: str, section_name: str) -> Optional[str]:
       # PASS 1: tolerant regex allowing single or double newlines
       pattern = (
           rf"(?:^|\n){{1,2}}\s*{re.escape(section_name)}\s*:\s*\n?"
           rf"(.*?)"
           rf"(?=(?:\n){{1,2}}\s*[A-Za-z-]+\s*:\s*\n|\Z)"
       )
       m = re.search(pattern, text, re.DOTALL | re.IGNORECASE)
       if m:
           return m.group(1).strip()

       # PASS 2: line-by-line scan (existing fallback, slightly improved)
       lines = text.split("\n")
       capturing = False
       captured = []
       for line in lines:
           stripped = line.strip()
           if re.match(rf"^{re.escape(section_name)}\s*:\s*$", stripped, re.IGNORECASE):
               capturing = True
               continue
           if capturing and re.match(r"^[A-Za-z-]+\s*:\s*$", stripped):
               break
           if capturing:
               captured.append(line)
       if captured:
           return "\n".join(captured).strip()
       return None

2. Add a v2 format as an opt-in alternative. The parser already checks for
   `BDA Submission v1`. Add `BDA Submission v2` that uses YAML frontmatter:

   BDA Submission v2
   ---
   Debate-ID: abc123
   Resolution: AI should be regulated
   Submission-ID: uuid-here
   Submitted-At: 2026-04-30T00:00:00Z
   Position: FOR
   Topic-Area: t1
   ---

   Facts:
   - Fact one
   - Fact two

   Inference:
   The conclusion follows.

   Counter-Arguments:
   Optional rebuttal.

   The v2 header block is delimited by `---` and can be parsed with the
   standard PyYAML parser or a lightweight custom parser. This removes all
   ambiguity around newlines.

3. Update parse_body() to dispatch based on the header line:

   if self.HEADER_RE.match(lines[0].strip()):
       return self._parse_v1(lines)
   elif self.HEADER_V2_RE.match(lines[0].strip()):
       return self._parse_v2(text)
   else:
       raise EmailSubmissionError("Unrecognized submission format.")

4. Add tests:
   • test_extract_section_tolerates_single_newlines
   • test_extract_section_tolerates_trailing_whitespace
   • test_v2_format_parses_correctly
   • test_v2_format_rejects_missing_frontmatter

ACCEPTANCE CRITERIA
-------------------
• An email with single newlines between sections still parses correctly.
• Trailing whitespace after "Facts:" or "Inference:" does not break extraction.
• The v2 YAML-frontmatter format is accepted as a valid alternative.
• All existing v1 tests continue to pass (backward compatibility).
